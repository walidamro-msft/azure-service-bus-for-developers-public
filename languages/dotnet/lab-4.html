Lab 4 - Diagnose and recover poison messages

Create an invalid order, dead-letter it with diagnostics, repair it, and re-drive it safely.

60 minutesThree console appsDead-letter queue

Case study: isolate an invalid order

A malformed order must not block valid work or disappear without evidence. The validator moves it to the queue's dead-letter subqueue with a reason and description. A recovery operator inspects that metadata, creates a corrected copy, and completes the original only after the replacement is accepted.

flowchart LR
 P[Poison producer] -- invalid order --> Q[orders queue]
 Q -- validation failure --> D[orders DLQ]
 D -- inspect and repair --> R[Recovery app]
 R -- corrected copy --> Q
Important: Service Bus does not automatically clean the DLQ. Production systems should alert on dead-letter counts and retain reason, description, and correlation identifiers.

2. Send an invalid order

Replace Contoso.Poison.Producer/Program.cs with:

using Azure;
using Azure.Messaging.ServiceBus;
using Microsoft.Extensions.Configuration;

// Load local Service Bus settings copied next to the compiled app.
IConfiguration configuration = new ConfigurationBuilder()
    .SetBasePath(AppContext.BaseDirectory)
    .AddJsonFile("appsettings.json", optional: false)
    .Build();

string fqns = configuration["ServiceBus:FullyQualifiedNamespace"]
    ?? throw new InvalidOperationException("Set ServiceBus:FullyQualifiedNamespace in appsettings.json.");
string sasKeyName = configuration["ServiceBus:SasKeyName"]
    ?? throw new InvalidOperationException("Set ServiceBus:SasKeyName in appsettings.json.");
string sasKey = configuration["ServiceBus:SasKey"]
    ?? throw new InvalidOperationException("Set ServiceBus:SasKey in appsettings.json.");
string queueName = configuration["ServiceBus:QueueName"]
    ?? throw new InvalidOperationException("Set ServiceBus:QueueName in appsettings.json.");

// ProducerSAS needs Send permission only because this app only publishes messages.
var credential = new AzureNamedKeyCredential(sasKeyName, sasKey);
await using var client = new ServiceBusClient(fqns, credential);
await using ServiceBusSender sender = client.CreateSender(queueName);

// Intentionally send plain text instead of the JSON order expected by the validator.
// This application-level poison message will fail validation and be moved to the DLQ.
await sender.SendMessageAsync(new ServiceBusMessage("not-json")
{
    MessageId = $"POISON-{DateTime.UtcNow:HHmmss}",
    Subject = "OrderCreated"
});
Console.WriteLine("Sent one invalid order.");

3. Validate and dead-letter

Replace Contoso.Order.Validator/Program.cs with:

using Azure;
using Azure.Messaging.ServiceBus;
using Microsoft.Extensions.Configuration;
using System.Text.Json;

// Load local Service Bus settings copied next to the compiled app.
IConfiguration configuration = new ConfigurationBuilder()
    .SetBasePath(AppContext.BaseDirectory)
    .AddJsonFile("appsettings.json", optional: false)
    .Build();

string fqns = configuration["ServiceBus:FullyQualifiedNamespace"]
    ?? throw new InvalidOperationException("Set ServiceBus:FullyQualifiedNamespace in appsettings.json.");
string sasKeyName = configuration["ServiceBus:SasKeyName"]
    ?? throw new InvalidOperationException("Set ServiceBus:SasKeyName in appsettings.json.");
string sasKey = configuration["ServiceBus:SasKey"]
    ?? throw new InvalidOperationException("Set ServiceBus:SasKey in appsettings.json.");
string queueName = configuration["ServiceBus:QueueName"]
    ?? throw new InvalidOperationException("Set ServiceBus:QueueName in appsettings.json.");

// ReceiverSAS needs Listen permission to receive and settle queue messages.
var credential = new AzureNamedKeyCredential(sasKeyName, sasKey);
await using var client = new ServiceBusClient(fqns, credential);
await using ServiceBusReceiver receiver = client.CreateReceiver(queueName);

// Receive in the default PeekLock mode so validation decides the final settlement.
ServiceBusReceivedMessage message = await receiver.ReceiveMessageAsync(TimeSpan.FromSeconds(20))
    ?? throw new InvalidOperationException("No order was available.");
try
{
    // The order is valid only when its body is JSON and contains an orderId property.
    using JsonDocument order = JsonDocument.Parse(message.Body);
    _ = order.RootElement.GetProperty("orderId").GetString();

    // Complete the valid message to remove it permanently from the queue.
    await receiver.CompleteMessageAsync(message);
    Console.WriteLine("Validated and completed the order.");
}
catch (Exception exception) when (exception is JsonException or KeyNotFoundException)
{
    // Move the poison message to the DLQ and preserve diagnostics for recovery.
    await receiver.DeadLetterMessageAsync(message, "InvalidOrderJson", exception.Message);
    Console.WriteLine($"Dead-lettered {message.MessageId}: InvalidOrderJson");
}

4. Inspect, repair, and re-drive

Replace Contoso.Dlq.Recovery/Program.cs with:

using Azure;
using Azure.Messaging.ServiceBus;
using Microsoft.Extensions.Configuration;
using System.Text.Json;

// Load local Service Bus settings copied next to the compiled app.
IConfiguration configuration = new ConfigurationBuilder()
    .SetBasePath(AppContext.BaseDirectory)
    .AddJsonFile("appsettings.json", optional: false)
    .Build();

string fqns = configuration["ServiceBus:FullyQualifiedNamespace"]
    ?? throw new InvalidOperationException("Set ServiceBus:FullyQualifiedNamespace in appsettings.json.");
string sasKeyName = configuration["ServiceBus:SasKeyName"]
    ?? throw new InvalidOperationException("Set ServiceBus:SasKeyName in appsettings.json.");
string sasKey = configuration["ServiceBus:SasKey"]
    ?? throw new InvalidOperationException("Set ServiceBus:SasKey in appsettings.json.");
string queueName = configuration["ServiceBus:QueueName"]
    ?? throw new InvalidOperationException("Set ServiceBus:QueueName in appsettings.json.");

// RecoverySAS needs Listen to receive/complete from the DLQ and Send to re-drive.
var credential = new AzureNamedKeyCredential(sasKeyName, sasKey);
await using var client = new ServiceBusClient(fqns, credential);

// Open a receiver on the queue's dead-letter subqueue, not the active queue.
await using ServiceBusReceiver dlq = client.CreateReceiver(
    queueName, new ServiceBusReceiverOptions { SubQueue = SubQueue.DeadLetter });
await using ServiceBusSender sender = client.CreateSender(queueName);

// Lock one dead-lettered message while it is inspected and repaired.
ServiceBusReceivedMessage failed = await dlq.ReceiveMessageAsync(TimeSpan.FromSeconds(20))
    ?? throw new InvalidOperationException("The DLQ is empty.");

// Print dead-letter diagnostics captured by the validator.
Console.WriteLine($"Reason: {failed.DeadLetterReason}");
Console.WriteLine($"Description: {failed.DeadLetterErrorDescription}");

// Replace the malformed body with a valid order and retain traceability to the original.
string repairedBody = JsonSerializer.Serialize(new { orderId = "ORD-RECOVERED", product = "AZ-BOOK", quantity = 1 });
var repaired = new ServiceBusMessage(repairedBody)
{
    // Use a new MessageId because Service Bus messages cannot be modified in place.
    MessageId = $"{failed.MessageId}-repaired",
    CorrelationId = failed.MessageId,
    Subject = failed.Subject,
    ContentType = "application/json"
};

// Send the repaired copy before completing the original; if send fails, the locked
// DLQ message remains available for another recovery attempt instead of being lost.
await sender.SendMessageAsync(repaired);
await dlq.CompleteMessageAsync(failed);
Console.WriteLine($"Re-drove {repaired.MessageId} and completed the DLQ original.");

5. Run the recovery drill

dotnet run --project Contoso.Poison.Producer
dotnet run --project Contoso.Order.Validator
# Verify the portal shows one dead-lettered message.
dotnet run --project Contoso.Dlq.Recovery
# Run the validator again to consume the repaired copy.
dotnet run --project Contoso.Order.Validator

Expected result: The DLQ count rises to one after validation, then returns to zero after recovery. The final validator run completes ORD-RECOVERED, and the active queue count returns to zero.

Challenge: Change the recovery policy so unknown dead-letter reasons are abandoned for investigation rather than automatically re-driven.
