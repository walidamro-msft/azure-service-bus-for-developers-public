Lab 1 - Queue-based order intake

Create a Service Bus queue, then build separate producer and receiver console applications.

60 minutesTwo console appsQueues

Case study: protect checkout from traffic spikes

Contoso Shop must accept orders even when the fulfilment service is busy or temporarily offline. The checkout producer writes each order to a durable queue and returns immediately. A separate fulfilment receiver drains the queue at its own pace.

flowchart LR
  C[Checkout producer] -- OrderCreated --> Q[orders queue]
  Q -- competing consumer --> F[Fulfilment receiver]

Outcome: You will deploy the queue, build two independent console applications, send ten orders, and prove that the receiver settles every message.

1. Provision the queue

Use Standard so the same namespace can support topics and sessions in later labs. Basic is sufficient for a queue-only experiment; Premium is appropriate when production requires dedicated capacity, private endpoints, or predictable latency.

Option A: Azure portal

  1. Open the Create a Service Bus namespace page.
  2. Select your subscription, create or select rg-sb-workshop, enter a globally unique namespace such as sb-contoso-<unique>, choose a region, and select the Standard pricing tier.
  3. Select Review + create, then Create. When deployment finishes, open the namespace.
  4. Under Entities, select Queues and + Queue. Set the name to orders, set Max delivery count to 5, keep the other defaults, and select Create.
  5. On the namespace Overview page, select the Local Authentication status, set it to Enabled, and save the change. If the Shared access policies page displays a warning that SAS key authentication is disabled, select the link in that warning to open the same setting.
  6. Under Settings, open Shared access policies, select + Add, create a policy named ProducerSAS, select only Send, and select Create. Do not select Listen or Manage.
  7. Select + Add again, create a policy named ReceiverSAS, select only Listen, and select Create. Do not select Send or Manage.
  8. Open ProducerSAS and copy its Primary key for the producer settings. Then open ReceiverSAS and copy its Primary key for the receiver settings.
  9. On the namespace Overview page, copy the hostname ending in .servicebus.windows.net. Use only the FQDN, for example sb-contoso-12345.servicebus.windows.net: do not include https://, a trailing slash, or the queue name.

Option B: Azure CLI

az login
$suffix = Get-Random -Maximum 99999
$resourceGroup = "rg-sb-workshop"
$namespace = "sb-contoso-$suffix"

az group create --name $resourceGroup --location eastus
az servicebus namespace create --resource-group $resourceGroup --name $namespace --location eastus --sku Standard
az servicebus namespace update --resource-group $resourceGroup --name $namespace --disable-local-auth false
az servicebus queue create --resource-group $resourceGroup --namespace-name $namespace --name orders --max-delivery-count 5

az servicebus namespace authorization-rule create --resource-group $resourceGroup --namespace-name $namespace --name ProducerSAS --rights Send
az servicebus namespace authorization-rule create --resource-group $resourceGroup --namespace-name $namespace --name ReceiverSAS --rights Listen
$producerSasKey = az servicebus namespace authorization-rule keys list --resource-group $resourceGroup --namespace-name $namespace --name ProducerSAS --query primaryKey -o tsv
$receiverSasKey = az servicebus namespace authorization-rule keys list --resource-group $resourceGroup --namespace-name $namespace --name ReceiverSAS --query primaryKey -o tsv

"Producer settings"
[PSCustomObject]@{
  FullyQualifiedNamespace = "$namespace.servicebus.windows.net"
  SasKeyName = "ProducerSAS"
  SasKey = $producerSasKey
  QueueName = "orders"
} | Format-List

"Receiver settings"
[PSCustomObject]@{
  FullyQualifiedNamespace = "$namespace.servicebus.windows.net"
  SasKeyName = "ReceiverSAS"
  SasKey = $receiverSasKey
  QueueName = "orders"
} | Format-List

Copy the first four displayed values into the producer's appsettings.json and the second four into the receiver's file.

Important: SAS authentication works only when namespace local authentication is enabled. Complete either Option A or Option B, not both. Treat the SAS key as a secret and do not commit either appsettings.json file to source control.

2. Create two console applications

mkdir lab1-queue
cd lab1-queue
dotnet new sln --name Contoso.QueueLab
dotnet new console --name Contoso.Checkout.Producer
dotnet new console --name Contoso.Fulfilment.Receiver

dotnet add Contoso.Fulfilment.Receiver package Microsoft.Extensions.Configuration.Json

Producer settings

Create Contoso.Checkout.Producer/appsettings.json with the values copied during provisioning:

{
  "ServiceBus": {
    "FullyQualifiedNamespace": "sb-contoso-12345.servicebus.windows.net",
    "SasKeyName": "ProducerSAS",
    "SasKey": "<ProducerSAS-primary-key>",
    "QueueName": "orders"
  }
}

Use this appsettings.json file as the configuration source for the producer in this lab. Do not configure these values through environment variables.

Receiver settings

Create Contoso.Fulfilment.Receiver/appsettings.json with the receiver policy and key:

{
  "ServiceBus": {
    "FullyQualifiedNamespace": "sb-contoso-12345.servicebus.windows.net",
    "SasKeyName": "ReceiverSAS",
    "SasKey": "<ReceiverSAS-primary-key>",
    "QueueName": "orders"
  }
}

In both project files, add this item group inside the root <Project> element so appsettings.json is copied beside the compiled application:

<ItemGroup>
  <None Update="appsettings.json">
    <CopyToOutputDirectory>PreserveNewest</CopyToOutputDirectory>
  </None>
</ItemGroup>

The producer and receiver are separate processes. ProducerSAS can only send messages, while ReceiverSAS can only receive and settle messages. Neither application requires the Manage claim. The FQDN selects the namespace, while QueueName is passed to CreateSender or CreateProcessor to select the orders queue inside that namespace.

Important: Both settings files contain the SAS key. Keep them local for this lab and add Contoso.Checkout.Producer/appsettings.json and Contoso.Fulfilment.Receiver/appsettings.json to your repository's .gitignore before committing any project files. This lab uses appsettings.json only and intentionally avoids environment-variable configuration.

3. Build the checkout producer

Replace Contoso.Checkout.Producer/Program.cs with:

using Azure;
using Azure.Messaging.ServiceBus;
using Microsoft.Extensions.Configuration;
using System.Text.Json;

// Load the settings file copied beside the compiled application.
IConfiguration configuration = new ConfigurationBuilder()
  .SetBasePath(AppContext.BaseDirectory)
  .AddJsonFile("appsettings.json", optional: false)
  .Build();

string fqns = configuration["ServiceBus:FullyQualifiedNamespace"]
  ?? throw new InvalidOperationException("Set ServiceBus:FullyQualifiedNamespace in appsettings.json.");
string sasKeyName = configuration["ServiceBus:SasKeyName"]
  ?? throw new InvalidOperationException("Set ServiceBus:SasKeyName in appsettings.json.");
string sasKey = configuration["ServiceBus:SasKey"]
  ?? throw new InvalidOperationException("Set ServiceBus:SasKey in appsettings.json.");
string queueName = configuration["ServiceBus:QueueName"]
  ?? throw new InvalidOperationException("Set ServiceBus:QueueName in appsettings.json.");

// Authenticate with the send-only ProducerSAS policy.
var credential = new AzureNamedKeyCredential(sasKeyName, sasKey);
await using var client = new ServiceBusClient(fqns, credential);

// The sender targets the queue named in appsettings.json within this namespace.
await using ServiceBusSender sender = client.CreateSender(queueName);

for (int number = 1; number <= 10; number++)
{
    string orderId = $"ORD-{number:000}";
    string body = JsonSerializer.Serialize(new { orderId, product = "AZ-BOOK", quantity = number });

    // MessageId supports tracing and Subject identifies the business event type.
    var message = new ServiceBusMessage(body)
    {
        MessageId = orderId,
        Subject = "OrderCreated",
        ContentType = "application/json"
    };

    await sender.SendMessageAsync(message);
    Console.WriteLine($"Sent {orderId}");
}

4. Build the fulfilment receiver

Replace Contoso.Fulfilment.Receiver/Program.cs with:

using Azure;
using Azure.Messaging.ServiceBus;
using Microsoft.Extensions.Configuration;
using System.Text.Json;

// Load the settings file copied beside the compiled application.
IConfiguration configuration = new ConfigurationBuilder()
  .SetBasePath(AppContext.BaseDirectory)
  .AddJsonFile("appsettings.json", optional: false)
  .Build();

string fqns = configuration["ServiceBus:FullyQualifiedNamespace"]
  ?? throw new InvalidOperationException("Set ServiceBus:FullyQualifiedNamespace in appsettings.json.");
string sasKeyName = configuration["ServiceBus:SasKeyName"]
  ?? throw new InvalidOperationException("Set ServiceBus:SasKeyName in appsettings.json.");
string sasKey = configuration["ServiceBus:SasKey"]
  ?? throw new InvalidOperationException("Set ServiceBus:SasKey in appsettings.json.");
string queueName = configuration["ServiceBus:QueueName"]
  ?? throw new InvalidOperationException("Set ServiceBus:QueueName in appsettings.json.");

// Authenticate with the listen-only ReceiverSAS policy.
var credential = new AzureNamedKeyCredential(sasKeyName, sasKey);
await using var client = new ServiceBusClient(fqns, credential);

// Allow the processor to handle up to two messages concurrently.
ServiceBusProcessor processor = client.CreateProcessor(queueName, new ServiceBusProcessorOptions
{
    AutoCompleteMessages = false,
    MaxConcurrentCalls = 2
});

processor.ProcessMessageAsync += async args =>
{
    using JsonDocument order = JsonDocument.Parse(args.Message.Body);
    Console.WriteLine($"Received {order.RootElement.GetProperty("orderId").GetString()} " +
                      $"(delivery {args.Message.DeliveryCount})");

    // Explicit settlement removes the message only after processing succeeds.
    await args.CompleteMessageAsync(args.Message);
};

// Processor-level failures, such as connection errors, arrive on this callback.
processor.ProcessErrorAsync += args =>
{
    Console.Error.WriteLine(args.Exception.Message);
    return Task.CompletedTask;
};

await processor.StartProcessingAsync();
Console.WriteLine("Receiver is running. Press Enter to stop.");
Console.ReadLine();
await processor.StopProcessingAsync();

5. Run and verify

After saving both appsettings.json files, open two terminals and run:

# Terminal 1: start the receiver
dotnet run --project Contoso.Fulfilment.Receiver

# Terminal 2: send ten orders
dotnet run --project Contoso.Checkout.Producer

Expected result: The producer prints ten sent order IDs. The receiver prints those ten IDs and completes them. In the Azure portal, the queue's active message count returns to zero.

Challenge: Stop the receiver, run the producer again, and inspect the queue count. Restart the receiver and observe the durable backlog drain. Then run two receiver instances and confirm each order is processed by only one competing consumer.
