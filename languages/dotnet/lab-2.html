Lab 2 - Publish one event to two subscribers

Create a topic, build a publisher console app, and run a subscriber app for two independent subscriptions.

60 minutesThree processesTopics

Case study: notify inventory and customer care

When checkout confirms an order, inventory must reserve stock and customer care must prepare a confirmation. The publisher sends one OrderCreated event. Service Bus copies it into two durable subscriptions so both business capabilities receive the event independently.

flowchart LR
  P[Order publisher] -- one message --> T[order-events topic]
  T -- copy --> I[inventory subscription]
  T -- copy --> C[customer-care subscription]
  I -- receive --> SI[Subscriber instance: inventory]
  C -- receive --> SC[Subscriber instance: customer-care]
Important: Run each receiver against a different subscription. Two receivers using the same subscription are competing consumers and would split messages instead of both receiving a copy.

1. Provision the topic and subscriptions

Reuse the namespace and PowerShell variables from Lab 1.

Option A: Azure portal

  1. In the Azure portal, open the Service Bus namespace created in Lab 1.
  2. Under Entities, select Topics, select + Topic, enter order-events, keep the defaults, and select Create.
  3. Open the order-events topic. Under Entities, select Subscriptions, then select + Subscription.
  4. Create a subscription named inventory and keep the default settings.
  5. Repeat the preceding step to create a second subscription named customer-care.
  6. Confirm that both subscriptions appear under the topic. You will use this namespace hostname in appsettings.json for both applications.

Option B: Azure CLI

az servicebus topic create --resource-group $resourceGroup --namespace-name $namespace --name order-events
az servicebus topic subscription create --resource-group $resourceGroup --namespace-name $namespace --topic-name order-events --name inventory
az servicebus topic subscription create --resource-group $resourceGroup --namespace-name $namespace --topic-name order-events --name customer-care

Checkpoint: Complete either Option A or Option B, not both. The portal shows one topic and two subscriptions, each with an active message count of zero.

2. Create publisher and subscriber applications

mkdir lab2-topic
cd lab2-topic
dotnet new sln --name Contoso.TopicLab
dotnet new console --name Contoso.Orders.Publisher
dotnet new console --name Contoso.Events.Subscriber
dotnet sln add Contoso.Orders.Publisher Contoso.Events.Subscriber

dotnet add Contoso.Orders.Publisher package Azure.Messaging.ServiceBus
dotnet add Contoso.Orders.Publisher package Microsoft.Extensions.Configuration.Json
dotnet add Contoso.Events.Subscriber package Azure.Messaging.ServiceBus
dotnet add Contoso.Events.Subscriber package Microsoft.Extensions.Configuration.Json

Publisher settings

Create Contoso.Orders.Publisher/appsettings.json:

{
  "ServiceBus": {
    "FullyQualifiedNamespace": "sb-contoso-12345.servicebus.windows.net",
    "SasKeyName": "ProducerSAS",
    "SasKey": "<ProducerSAS-primary-key>",
    "TopicName": "order-events"
  }
}

Subscriber settings

Create Contoso.Events.Subscriber/appsettings.json:

{
  "ServiceBus": {
    "FullyQualifiedNamespace": "sb-contoso-12345.servicebus.windows.net",
    "SasKeyName": "ReceiverSAS",
    "SasKey": "<ReceiverSAS-primary-key>",
    "TopicName": "order-events"
  }
}

In both project files, add this item group inside the root <Project> element so appsettings.json is copied beside the compiled application:

<ItemGroup>
  <None Update="appsettings.json">
    <CopyToOutputDirectory>PreserveNewest</CopyToOutputDirectory>
  </None>
</ItemGroup>
Important: This lab uses appsettings.json for Service Bus configuration and does not use environment variables. Keep both settings files local and add Contoso.Orders.Publisher/appsettings.json and Contoso.Events.Subscriber/appsettings.json to .gitignore.

The subscriber accepts a subscription name as a command-line argument, so the same application can run as two independent business consumers.

3. Build the order publisher

Replace Contoso.Orders.Publisher/Program.cs with:

using Azure;
using Azure.Messaging.ServiceBus;
using Microsoft.Extensions.Configuration;
using System.Text.Json;

IConfiguration configuration = new ConfigurationBuilder()
  .SetBasePath(AppContext.BaseDirectory)
  .AddJsonFile("appsettings.json", optional: false)
  .Build();

string fqns = configuration["ServiceBus:FullyQualifiedNamespace"]
  ?? throw new InvalidOperationException("Set ServiceBus:FullyQualifiedNamespace in appsettings.json.");
string sasKeyName = configuration["ServiceBus:SasKeyName"]
  ?? throw new InvalidOperationException("Set ServiceBus:SasKeyName in appsettings.json.");
string sasKey = configuration["ServiceBus:SasKey"]
  ?? throw new InvalidOperationException("Set ServiceBus:SasKey in appsettings.json.");
string topicName = configuration["ServiceBus:TopicName"]
  ?? throw new InvalidOperationException("Set ServiceBus:TopicName in appsettings.json.");

var credential = new AzureNamedKeyCredential(sasKeyName, sasKey);
await using var client = new ServiceBusClient(fqns, credential);
await using ServiceBusSender sender = client.CreateSender(topicName);

string orderId = $"ORD-{DateTime.UtcNow:HHmmss}";
var message = new ServiceBusMessage(JsonSerializer.Serialize(new
{
    orderId,
    product = "AZ-BOOK",
    quantity = 2
}))
{
    MessageId = orderId,
    Subject = "OrderCreated",
    ContentType = "application/json"
};

await sender.SendMessageAsync(message);
Console.WriteLine($"Published one event for {orderId}");

4. Build the reusable subscriber

Replace Contoso.Events.Subscriber/Program.cs with:

using Azure;
using Azure.Messaging.ServiceBus;
using Microsoft.Extensions.Configuration;

if (args.Length != 1)
    throw new ArgumentException("Pass inventory or customer-care.");

string subscriptionName = args[0];
IConfiguration configuration = new ConfigurationBuilder()
  .SetBasePath(AppContext.BaseDirectory)
  .AddJsonFile("appsettings.json", optional: false)
  .Build();

string fqns = configuration["ServiceBus:FullyQualifiedNamespace"]
  ?? throw new InvalidOperationException("Set ServiceBus:FullyQualifiedNamespace in appsettings.json.");
string sasKeyName = configuration["ServiceBus:SasKeyName"]
  ?? throw new InvalidOperationException("Set ServiceBus:SasKeyName in appsettings.json.");
string sasKey = configuration["ServiceBus:SasKey"]
  ?? throw new InvalidOperationException("Set ServiceBus:SasKey in appsettings.json.");
string topicName = configuration["ServiceBus:TopicName"]
  ?? throw new InvalidOperationException("Set ServiceBus:TopicName in appsettings.json.");

var credential = new AzureNamedKeyCredential(sasKeyName, sasKey);
await using var client = new ServiceBusClient(fqns, credential);
ServiceBusProcessor processor = client.CreateProcessor(
  topicName,
    subscriptionName,
    new ServiceBusProcessorOptions { AutoCompleteMessages = false });

processor.ProcessMessageAsync += async eventArgs =>
{
    Console.WriteLine($"[{subscriptionName}] received {eventArgs.Message.MessageId}: {eventArgs.Message.Body}");
    await eventArgs.CompleteMessageAsync(eventArgs.Message);
};
processor.ProcessErrorAsync += eventArgs =>
{
    Console.Error.WriteLine($"[{subscriptionName}] {eventArgs.Exception.Message}");
    return Task.CompletedTask;
};

await processor.StartProcessingAsync();
Console.WriteLine($"Listening on {subscriptionName}. Press Enter to stop.");
Console.ReadLine();
await processor.StopProcessingAsync();

5. Run three processes and verify fan-out

# Terminal 1
dotnet run --project Contoso.Events.Subscriber -- inventory

# Terminal 2
dotnet run --project Contoso.Events.Subscriber -- customer-care

# Terminal 3
dotnet run --project Contoso.Orders.Publisher

Expected result: The publisher reports one sent event. Both subscriber terminals display the same MessageId and body. Each subscription's active message count returns to zero.

Challenge: Stop only the customer-care process and publish another event. Inventory processes it immediately while the customer-care subscription retains its own copy. Restart customer-care and observe it receive the stored event.
