Lab 3 - Preserve each order's event sequence

Provision a session-enabled queue, send interleaved workflows, and process each order in FIFO order.

60 minutesTwo console appsSessions

Case study: ordered fulfilment workflows

Payment, allocation, and dispatch events for many orders may share one queue. Events from different orders can run concurrently, but no order may dispatch before its own payment and allocation steps. A Service Bus session groups each order's messages under its order ID.

flowchart LR
  S[Workflow producer] -- SessionId = order ID --> Q[order-sessions queue]
  Q -- one locked session --> W1[Worker]
  Q -- another locked session --> W2[Worker]

1. Provision a session-enabled queue

Reuse the namespace from Lab 1. Session support must be enabled when the queue is created.

Option A: Azure portal

  1. In the Azure portal, open the Service Bus namespace created in Lab 1.
  2. Under Entities, select Queues, then select + Queue.
  3. Enter order-sessions, enable Sessions, and set Max delivery count to 5.
  4. Select Create, open the queue, and verify that Requires session is enabled.
  5. Copy the namespace hostname from Lab 1 and use it in each application's appsettings.json.
Important: You cannot enable or disable sessions after a queue is created. Delete and recreate an incorrectly configured queue before sending messages.

Option B: Azure CLI

az servicebus queue create --resource-group $resourceGroup --namespace-name $namespace --name order-sessions --enable-session true --max-delivery-count 5

Checkpoint: Complete either Option A or Option B, not both. The portal shows order-sessions with sessions enabled and zero active messages.

3. Send two interleaved order workflows

Replace Contoso.Workflow.Producer/Program.cs with:

Producer settings

Create Contoso.Workflow.Producer/appsettings.json:

{
    "ServiceBus": {
        "FullyQualifiedNamespace": "sb-contoso-12345.servicebus.windows.net",
        "SasKeyName": "ProducerSAS",
        "SasKey": "<ProducerSAS-primary-key>",
        "QueueName": "order-sessions"
    }
}

Producer code

using Azure;
using Azure.Messaging.ServiceBus;
using Microsoft.Extensions.Configuration;

// Load appsettings.json from the output directory.
IConfiguration configuration = new ConfigurationBuilder()
        .SetBasePath(AppContext.BaseDirectory)
        .AddJsonFile("appsettings.json", optional: false)
        .Build();

string fqns = configuration["ServiceBus:FullyQualifiedNamespace"]
        ?? throw new InvalidOperationException("Set ServiceBus:FullyQualifiedNamespace in appsettings.json.");
string sasKeyName = configuration["ServiceBus:SasKeyName"]
        ?? throw new InvalidOperationException("Set ServiceBus:SasKeyName in appsettings.json.");
string sasKey = configuration["ServiceBus:SasKey"]
        ?? throw new InvalidOperationException("Set ServiceBus:SasKey in appsettings.json.");
string queueName = configuration["ServiceBus:QueueName"]
        ?? throw new InvalidOperationException("Set ServiceBus:QueueName in appsettings.json.");

// Authenticate with the send-only SAS policy and target the session-enabled queue.
var credential = new AzureNamedKeyCredential(sasKeyName, sasKey);
await using var client = new ServiceBusClient(fqns, credential);
await using ServiceBusSender sender = client.CreateSender(queueName);

string[] orderIds = ["ORD-101", "ORD-202"];
string[] states = ["PaymentAccepted", "StockAllocated", "Dispatched"];
foreach (string state in states)
{
    foreach (string orderId in orderIds)
    {
        // SessionId groups all messages for one order so that order is processed sequentially.
        await sender.SendMessageAsync(new ServiceBusMessage(state)
        {
            SessionId = orderId,
            MessageId = $"{orderId}-{state}",
            Subject = state
        });
        Console.WriteLine($"Sent {orderId}: {state}");
    }
}

4. Process sessions concurrently

Create Contoso.Workflow.Processor/appsettings.json with the listen-only policy:

{
    "ServiceBus": {
        "FullyQualifiedNamespace": "sb-contoso-12345.servicebus.windows.net",
        "SasKeyName": "ReceiverSAS",
        "SasKey": "<ReceiverSAS-primary-key>",
        "QueueName": "order-sessions"
    }
}

In both project files, add this item group inside the root <Project> element so appsettings.json is copied beside the compiled application:

<ItemGroup>
    <None Update="appsettings.json">
        <CopyToOutputDirectory>PreserveNewest</CopyToOutputDirectory>
    </None>
</ItemGroup>
Important: This lab uses appsettings.json for Service Bus configuration and does not use environment variables. Keep both settings files local and add Contoso.Workflow.Producer/appsettings.json and Contoso.Workflow.Processor/appsettings.json to .gitignore.

Replace Contoso.Workflow.Processor/Program.cs with:

using Azure;
using Azure.Messaging.ServiceBus;
using Microsoft.Extensions.Configuration;

// Load appsettings.json from the output directory.
IConfiguration configuration = new ConfigurationBuilder()
        .SetBasePath(AppContext.BaseDirectory)
        .AddJsonFile("appsettings.json", optional: false)
        .Build();

string fqns = configuration["ServiceBus:FullyQualifiedNamespace"]
        ?? throw new InvalidOperationException("Set ServiceBus:FullyQualifiedNamespace in appsettings.json.");
string sasKeyName = configuration["ServiceBus:SasKeyName"]
        ?? throw new InvalidOperationException("Set ServiceBus:SasKeyName in appsettings.json.");
string sasKey = configuration["ServiceBus:SasKey"]
        ?? throw new InvalidOperationException("Set ServiceBus:SasKey in appsettings.json.");
string queueName = configuration["ServiceBus:QueueName"]
        ?? throw new InvalidOperationException("Set ServiceBus:QueueName in appsettings.json.");

// Authenticate with the listen-only SAS policy.
var credential = new AzureNamedKeyCredential(sasKeyName, sasKey);
await using var client = new ServiceBusClient(fqns, credential);
ServiceBusSessionProcessor processor = client.CreateSessionProcessor(
        queueName,
    new ServiceBusSessionProcessorOptions
    {
        AutoCompleteMessages = false,
        // Process multiple sessions in parallel, but keep FIFO within each session.
        MaxConcurrentSessions = 2,
        MaxConcurrentCallsPerSession = 1
    });

processor.ProcessMessageAsync += async args =>
{
    // Messages inside one SessionId are delivered in order.
    Console.WriteLine($"{args.Message.SessionId}: {args.Message.Body}");
    await Task.Delay(300);
    // Complete only after processing succeeds.
    await args.CompleteMessageAsync(args.Message);
};
processor.ProcessErrorAsync += args =>
{
    Console.Error.WriteLine(args.Exception.Message);
    return Task.CompletedTask;
};

await processor.StartProcessingAsync();
Console.WriteLine("Session processor is running. Press Enter to stop.");
Console.ReadLine();
await processor.StopProcessingAsync();

5. Run and verify ordering

# Terminal 1
dotnet run --project Contoso.Workflow.Processor

# Terminal 2
dotnet run --project Contoso.Workflow.Producer

Expected result: Output for ORD-101 and ORD-202 may interleave, but each order always prints PaymentAccepted, StockAllocated, then Dispatched. The queue returns to zero active messages.

Challenge: Add a third order and increase MaxConcurrentSessions. Then set it to one and compare throughput without changing per-order ordering.
