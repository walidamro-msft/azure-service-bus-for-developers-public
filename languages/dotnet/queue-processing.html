Working with queues in .NET

Map Service Bus processing decisions to processors, receivers, options, settlement APIs, sessions, and client lifetimes in Azure.Messaging.ServiceBus.

C# / .NET 8+Azure.Messaging.ServiceBusProcessor and receiver models

Package, authentication, and connection

Install Azure.Messaging.ServiceBus and Azure.Identity. Use DefaultAzureCredential locally and managed identity for Azure-hosted workloads. Grant producers Azure Service Bus Data Sender and consumers Azure Service Bus Data Receiver on the required namespace or queue.

dotnet add package Azure.Messaging.ServiceBus
dotnet add package Azure.Identity
using Azure.Identity;
using Azure.Messaging.ServiceBus;

string fqns = Environment.GetEnvironmentVariable("SERVICEBUS_FQNS")
    ?? throw new InvalidOperationException("Set SERVICEBUS_FQNS.");

await using var client = new ServiceBusClient(fqns, new DefaultAzureCredential());
Production identity: DefaultAzureCredential can select a workload or managed identity without changing application code. Do not embed connection strings or keys.

Send to a queue

Create one ServiceBusSender per destination and reuse it. ServiceBusMessage holds the body and sender-assigned envelope fields; ServiceBusReceivedMessage adds broker delivery metadata on receive.

await using ServiceBusSender sender = client.CreateSender("orders-processing");

var message = new ServiceBusMessage(BinaryData.FromObjectAsJson(new
{
    orderId = "ORD-123",
    status = "created"
}))
{
    MessageId = Guid.NewGuid().ToString(),
    CorrelationId = "order:ORD-123",
    Subject = "OrderCreated.v1",
    ContentType = "application/json",
    TimeToLive = TimeSpan.FromMinutes(30)
};
message.ApplicationProperties["schemaVersion"] = 1;

await sender.SendMessageAsync(message);

For multiple messages, use CreateMessageBatchAsync and check TryAddMessage. A collection with too much encoded data is rejected even when each message fits individually.

.NET message objects and fields

Body

TypeBinaryData
Set onServiceBusMessage
Meaning

The opaque business payload. Construct from UTF-8 text, bytes, streams, or BinaryData.FromObjectAsJson.

Use

Set ContentType, version the contract, and deserialize only after validating the expected type and schema.

MessageId

Typestring
Limit128 characters
Meaning

Identity of one logical message and the key used by broker duplicate detection.

Use

Generate once and persist with an outbox record. Reuse it when retrying the same send; create a new value for a new event or command.

CorrelationId and Subject

Typestring
PurposeWorkflow and contract
Meaning

CorrelationId connects related operations. Subject names the business contract; older APIs call it Label.

Use

Propagate a workflow value such as order:ORD-123 and use a stable contract name such as OrderCreated.v1.

ContentType and ApplicationProperties

Typesstring, IDictionary<string, object>
PurposeFormat and routing metadata
Meaning

ContentType describes serialization. Application properties contain compact scalar metadata available to filters and receivers.

Use

Use MIME values such as application/json. Put schema version, tenant, or region in application properties, never secrets or payload fragments.

SessionId, PartitionKey, and TimeToLive

Typesstring, TimeSpan
PurposeOrdering, affinity, expiration
Meaning

SessionId selects an ordered session; PartitionKey selects partition affinity; TimeToLive sets a message-specific lifetime.

Use

Every message to a session queue needs SessionId. When both IDs are set they must match. TTL can't effectively exceed the queue default.

Received-only fields

ObjectServiceBusReceivedMessage
OwnerService Bus
Fields

SequenceNumber, EnqueuedTime, ExpiresAt, DeliveryCount, LockToken, LockedUntil, DeadLetterReason, and DeadLetterErrorDescription.

Use

Use them for settlement, lock monitoring, DLQ diagnosis, and operational telemetry. Do not use sequence number as a cross-queue business ID.

Continuous processing with ServiceBusProcessor

The processor owns the background receive loop, dispatches callbacks, renews locks, and reports infrastructure failures through the error callback. Register both handlers before starting it.

await using ServiceBusProcessor processor = client.CreateProcessor("orders", new ServiceBusProcessorOptions
{
    AutoCompleteMessages = false,
    MaxConcurrentCalls = 4,
    MaxAutoLockRenewalDuration = TimeSpan.FromMinutes(5),
    PrefetchCount = 0,
    ReceiveMode = ServiceBusReceiveMode.PeekLock,
    Identifier = $"orders-worker:{Environment.MachineName}"
});

processor.ProcessMessageAsync += async args =>
{
    try
    {
        await ProcessOrderAsync(args.Message.Body.ToString());
        await args.CompleteMessageAsync(args.Message);
    }
    catch (PermanentOrderException exception)
    {
        await args.DeadLetterMessageAsync(args.Message, "InvalidOrder", exception.Message);
    }
    catch
    {
        await args.AbandonMessageAsync(args.Message);
    }
};

processor.ProcessErrorAsync += args =>
{
    Console.Error.WriteLine($"{args.ErrorSource} {args.EntityPath}: {args.Exception}");
    return Task.CompletedTask;
};

await processor.StartProcessingAsync();
await shutdownSignal.Task;
await processor.StopProcessingAsync();

Do not assume the error handler represents one business message. It reports receive-pump, connection, lock, and callback infrastructure failures; settlement remains the responsibility of the message callback.

ServiceBusProcessorOptions mapping

AutoCompleteMessages

Data typebool
Defaulttrue
Behavior

Completes an unsettled message when the callback returns successfully. An exception abandons an unsettled message.

Strategy

Set false when code must explicitly choose complete, abandon, defer, or dead-letter. Every successful path must then settle or allow the lock to expire.

MaxConcurrentCalls

Data typeint
Default1
Behavior

Limits concurrent message callbacks on this processor instance.

Strategy

Potential concurrency is replicas multiplied by this value. Increase for independent I/O-bound work only while downstream capacity and lock completion remain healthy.

MaxAutoLockRenewalDuration

Data typeTimeSpan
Default5 minutes
Behavior

Sets the total period for automatic message-lock renewal while a callback runs. The entity's lock duration controls each individual renewal.

Strategy

Use a finite value above the longest expected successful handler time. Zero disables auto renewal; an infinite value can hide a stuck callback.

PrefetchCount

Data typeint
Default0
Behavior

Caches messages locally before callbacks request them. Locks are already running while messages wait in the cache.

Strategy

Increase gradually for fast, steady handlers. Keep it low for large messages, short locks, slow processing, or bursty traffic.

ReceiveMode

Data typeServiceBusReceiveMode
DefaultPeekLock
Behavior

PeekLock requires settlement. ReceiveAndDelete removes a message as it is transferred.

Strategy

Use PeekLock for business workloads. Use ReceiveAndDelete only when message loss after transfer is acceptable.

SubQueue

Data typeSubQueue
DefaultNone
Behavior

Selects the main entity, dead-letter subqueue, or transfer dead-letter subqueue.

Strategy

Use a separate processor or receiver with SubQueue.DeadLetter for intentional recovery. Keep the normal worker on None.

Identifier

Data typestring
DefaultGenerated unique value
Behavior

Adds a diagnostic processor name to selected logs and exceptions. It does not affect message identity or routing.

Strategy

Use a role plus unique replica suffix so operators can identify the failing processor instance.

Pull receiving with ServiceBusReceiver

Use a receiver when code must control receive timing, batch size, peeking, deferred retrieval, or a focused dead-letter workflow.

await using ServiceBusReceiver receiver = client.CreateReceiver("orders", new ServiceBusReceiverOptions
{
    ReceiveMode = ServiceBusReceiveMode.PeekLock,
    PrefetchCount = 0,
    Identifier = $"orders-batch:{Environment.MachineName}"
});

IReadOnlyList<ServiceBusReceivedMessage> messages =
    await receiver.ReceiveMessagesAsync(maxMessages: 20, maxWaitTime: TimeSpan.FromSeconds(5));

foreach (ServiceBusReceivedMessage message in messages)
{
    try
    {
        await ProcessOrderAsync(message.Body.ToString());
        await receiver.CompleteMessageAsync(message);
    }
    catch
    {
        await receiver.AbandonMessageAsync(message);
    }
}

ReceiveMessagesAsync returns up to the requested count and may return fewer. For long-running pull-based work, call RenewMessageLockAsync before expiry or design the work to finish within the entity lock duration.

Sessions, subscriptions, and dead-letter queues

  • For a topic subscription, call CreateProcessor(topicName, subscriptionName, options) or CreateReceiver(topicName, subscriptionName, options).
  • For continuous session processing, use CreateSessionProcessor and ServiceBusSessionProcessorOptions.
  • For one explicit session, call AcceptSessionAsync or AcceptNextSessionAsync and settle through the returned session receiver.
  • For dead-letter recovery, use a separate receiver with SubQueue = SubQueue.DeadLetter; inspect reason and description before resubmitting corrected work.
