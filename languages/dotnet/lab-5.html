Lab 5 - Scale orders from HTTP to storage

Build and test a complete serverless path through Functions, Service Bus, Event Grid, Logic Apps, and Table Storage.

90 minutesAzure FunctionsServerless integration

Case study: absorb a checkout surge

Flow: Azure Load Testing calls an HTTP-triggered .NET isolated Function. The Function sends an order to orders. When no receiver is active, Service Bus emits an Event Grid notification. A Logic App handles that notification, drains available messages with peek-lock, writes rows to Azure Table Storage, and completes each message.

flowchart LR
  ALT[Azure Load Testing] -- HTTP POST --> FN[.NET Function producer]
  FN -- managed identity --> SB[Service Bus orders queue]
  SB -- queue has messages event --> EG[Event Grid]
  EG -- trigger --> LA[Logic App consumer]
  LA -- receive and complete --> SB
  LA -- upsert order --> TS[Table Storage]

Outcome: Load-test an HTTP endpoint, buffer accepted orders, wake a serverless consumer, and verify that every unique order becomes a table row.

1. Provision the supporting resources

Reuse rg-sb-workshop, the Service Bus namespace, and the orders queue from Lab 1. Do not create another namespace or queue. If orders is missing, return to Lab 1 and create it before continuing.

Option A: Azure portal

  1. In the Azure portal, create a Storage account in rg-sb-workshop. Choose a unique name, the same region as the workshop namespace, Standard performance, and Locally-redundant storage (LRS).
  2. Open the storage account, select Storage browser > Tables > Add table, and create a table named Orders.
  3. Create a Function App in rg-sb-workshop. Select Consumption hosting, the same region, .NET 10 (isolated) as the runtime stack, and the storage account you just created. Enter a globally unique app name such as func-sb-<unique>.
  4. Open the Function App, select Settings > Identity, turn the system-assigned identity On, and save.
  5. Open the Lab 1 Service Bus namespace and copy its Host name from the Overview page, for example sb-contoso-12345.servicebus.windows.net. Do not include https://, a trailing slash, or /orders.
  6. Return to the Function App. Under Settings > Environment variables, add SERVICEBUS_FQNS and paste that Host name as its value. Add SERVICEBUS_QUEUE_NAME with the value orders, then apply the changes.
  7. Open the Service Bus namespace and select Access control (IAM) > Add role assignment. Assign Azure Service Bus Data Sender to the Function App's managed identity.

Replace the generated SubmitOrder.cs with this .NET 10 isolated ASP.NET Core integration implementation:

using System.Text.Json;
  using Azure.Messaging.ServiceBus;
  using Microsoft.AspNetCore.Http;
  using Microsoft.AspNetCore.Mvc;
  using Microsoft.Azure.Functions.Worker;
  using Microsoft.Extensions.Logging;

  namespace Contoso.OrderApi;

  public class SubmitOrder
{
    private readonly ServiceBusSender _sender;
    private readonly ILogger<SubmitOrder> _logger;

    public SubmitOrder(ServiceBusSender sender, ILogger<SubmitOrder> logger)
    {
      _sender = sender;
      _logger = logger;
    }

    [Function("SubmitOrder")]
    public async Task<IActionResult> Run(
      [HttpTrigger(AuthorizationLevel.Function, "post")] HttpRequest request)
    {
      using JsonDocument json = await JsonDocument.ParseAsync(request.Body);
      if (!json.RootElement.TryGetProperty("orderId", out JsonElement orderIdElement) ||
        string.IsNullOrWhiteSpace(orderIdElement.GetString()))
      {
        return new BadRequestObjectResult("Provide a non-empty orderId.");
      }

      string orderId = orderIdElement.GetString()!;
      string body = json.RootElement.GetRawText();
      await _sender.SendMessageAsync(new ServiceBusMessage(body)
      {
        MessageId = orderId,
        Subject = "OrderCreated",
        ContentType = "application/json"
      });

      _logger.LogInformation("Accepted order {OrderId}.", orderId);
      return new AcceptedResult(location: null, value: orderId);
    }
}

Replace Program.cs with the following code. It validates both settings at startup and creates one reusable sender for the configured queue:

using Azure.Identity;
  using Azure.Messaging.ServiceBus;
  using Microsoft.Azure.Functions.Worker;
  using Microsoft.Azure.Functions.Worker.Builder;
  using Microsoft.Extensions.DependencyInjection;
  using Microsoft.Extensions.Hosting;

  FunctionsApplicationBuilder builder = FunctionsApplication.CreateBuilder(args);
  builder.ConfigureFunctionsWebApplication();

  builder.Services
    .AddApplicationInsightsTelemetryWorkerService()
    .ConfigureFunctionsApplicationInsights();

  string fullyQualifiedNamespace =
    Environment.GetEnvironmentVariable("SERVICEBUS_FQNS")
    ?? throw new InvalidOperationException("Set SERVICEBUS_FQNS.");
  string queueName =
    Environment.GetEnvironmentVariable("SERVICEBUS_QUEUE_NAME")
    ?? throw new InvalidOperationException("Set SERVICEBUS_QUEUE_NAME.");

  builder.Services.AddSingleton(_ =>
    new ServiceBusClient(fullyQualifiedNamespace, new DefaultAzureCredential()));
  builder.Services.AddSingleton(provider =>
    provider.GetRequiredService<ServiceBusClient>().CreateSender(queueName));

  builder.Build().Run();

3. Configure Event Grid and the Logic App consumer

Event Grid does not receive or forward the Service Bus message body. It wakes the Logic App when messages are available and no receiver is listening; the Logic App then drains orders through the Service Bus connector.

Step 3.1: Create the Logic App

  1. In the Azure portal, select Create a resource, search for Logic App, and select Create.
  2. Choose the subscription used for the workshop and resource group rg-sb-workshop. Using the same region as the Service Bus namespace is recommended for lower latency, but Event Grid can deliver to a Logic App in another region. For Logic App name, use a descriptive name such as logic-sb-orders-consumer-<initials>. For example, logic-sb-orders-consumer-wa.
  3. For this workshop, select Consumption (Multi-tenant). It is the simplest pay-per-execution option and matches the navigation in the remaining steps. Standard is also supported; if you use it, create a Stateful workflow and select the managed Azure Event Grid connector from the Shared connector gallery.
  4. Select Review + create, then Create. If you chose Standard, open the new resource, create a Stateful workflow, and continue the lab inside that workflow's designer.
  5. Open the Logic App. Under Settings > Identity > System assigned, set Status to On and select Save. Copy the displayed Object (principal) ID for troubleshooting.
How to choose: Consumption contains one workflow and is recommended for this lab. Standard hosts one or more workflows and is useful when you need single-tenant hosting, higher throughput, private networking, or multiple workflows. Use a Stateful Standard workflow for this lab because it supports managed connector triggers and preserves run history.

Step 3.2: Grant the Logic App access

  1. Open the Lab 1 Service Bus namespace. Select Access control (IAM) > Add > Add role assignment.
  2. Select Azure Service Bus Data Receiver. For Assign access to, choose Managed identity, select Logic app, choose the Logic App created in Step 3.1, and complete the assignment.
  3. Open the storage account created in Step 1 and add another role assignment. Assign Storage Table Data Contributor to the same Logic App managed identity.
  4. Allow several minutes for both assignments to propagate before creating the managed-identity connections.
Checkpoint: On the Logic App's Identity > Azure role assignments page, verify that it has Service Bus Data Receiver on the namespace and Storage Table Data Contributor on the storage account.

Step 3.3: Add the Event Grid trigger

  1. Open the Logic App and select Development Tools > Logic app designer. Choose Blank Logic App.
  2. Search for Azure Event Grid and select the trigger When a resource event occurs. If prompted, create the Event Grid connection by signing in with your workshop Azure account.
  3. Set Subscription and Resource Group to the workshop values. Set Resource Type to Microsoft.ServiceBus.Namespaces and Resource Name to the Lab 1 namespace.
  4. For Event Type Item, select Microsoft.ServiceBus.ActiveMessagesAvailableWithNoListeners.
  5. Under Advanced parameters, add Suffix Filter and set it to /queues/orders. This limits the event subscription to the orders queue.
  6. Select Save. Saving creates the Event Grid event subscription on the Service Bus namespace; you do not need to create a separate system topic.
  7. Use the Azure portal search bar to open Event Grid Subscriptions. In the current resource-list view, use Subscription to select the workshop subscription and Resource Group to select rg-sb-workshop. Leave Location set to all. This view might not show a Topic Type filter. If the subscription appears, open it and confirm that the source resource is the Lab 1 Service Bus namespace and the endpoint is the Logic App.
Connection versus subscription: An azureeventgrid entry marked Connected under the Logic App's Connections page is the API connection used to authenticate the designer. It does not confirm that the trigger created an Event Grid event subscription. Verify the generated subscription separately in Event Grid Subscriptions or with the Azure CLI command below.

If the Event Grid subscription does not appear after applying the subscription and resource-group filters, check the following items:

  1. Confirm that you are editing either the Consumption Logic App's designer or an enabled Stateful workflow in a Standard Logic App. For Standard, use the Azure Event Grid managed connector from the Shared connector gallery.
  2. Return to the designer and open When a resource event occurs. Confirm that the Event Grid connection is signed in, the resource type is Microsoft.ServiceBus.Namespaces, the Lab 1 namespace is selected, and the event type is Microsoft.ServiceBus.ActiveMessagesAvailableWithNoListeners.
  3. Select Save and resolve any error shown by the designer. Publishing alone does not create the subscription when the trigger configuration is incomplete.
  4. Confirm that the signed-in account can create event subscriptions on the Service Bus namespace. Contributor, Owner, or EventGrid EventSubscription Contributor at the namespace or a parent scope provides this management-plane permission.
  5. Wait about a minute, then refresh Event Grid Subscriptions. Confirm that Subscription and Resource Group match the workshop and that Location is set to all.

The Logic App and Service Bus namespace can be in different regions; a region mismatch does not prevent creation of this webhook event subscription. You can also verify the subscription from Azure Cloud Shell:

$resourceGroup = "rg-sb-workshop"
$namespace = "<your-lab-1-namespace>"

  --resource-group $resourceGroup `

Important: Stop every console receiver before testing. Service Bus raises this event only when active messages are available and no receiver is actively listening.

Step 3.4: Build the queue-drain loop

  1. After the trigger, add Variables > Initialize variable. Name it queueEmpty, choose type Boolean, and set its initial value to false.
  2. Add Control > Until. Set its exit condition to @equals(variables('queueEmpty'), true). In Change limits, use a timeout of PT5M and a count of 500.
  3. Inside the Until loop, add the Azure Service Bus action Get messages from a queue (peek-lock). Create the connection with Authentication type: Managed Identity. For Namespace Endpoint, enter sb://<namespace>.servicebus.windows.net/. Choose queue orders and set Maximum message count to 20.
  4. Add a Condition. In its expression editor, use @empty(body('Get_messages_from_a_queue_(peek-lock)')). If your action was renamed, insert its Body from dynamic content and apply the empty function instead of typing the generated action name.
  5. In the True branch, add Set variable and set queueEmpty to true.
  6. In the False branch, add Control > For each. For its input, select Body from Get messages from a queue (peek-lock). The connector returns an array even when the batch contains one message.
  7. Inside For each, add Data Operations > Parse JSON. For Content, use @base64ToString(item()?['ContentData']). Generate the schema from {"orderId":"ORDER-001","amount":125}.

Step 3.5: Persist and settle each message

  1. Still inside For each, add Azure Table Storage > Insert or Replace Entity (V2). Do not select the deprecated action because managed identity works only with V2 Table actions.
  2. Create the Table connection with Authentication type: Managed Identity. For the storage endpoint, use https://<storage-account>.table.core.windows.net/; you can copy it from the storage account's Settings > Endpoints > Table service.
  3. Select table Orders. Set Partition Key to orders, set Row Key to the parsed orderId, and map the remaining parsed order fields into the entity.
  4. After the table action and still inside For each, add Azure Service Bus > Complete the message in a queue. Reuse the managed-identity connection, select orders, and set Lock token of the message to @item()?['LockToken'].
  5. Open Configure run after for the complete action and leave only is successful selected for the table write. A failed storage operation must not complete the message.
  6. Select Save. The designer should show no invalid parameters or disconnected actions.
Checkpoint: The successful path is Event Grid trigger → initialize queueEmpty → Until → peek-lock batch receive → For each → decode and parse → V2 table upsert → complete. The empty receive path sets queueEmpty to true and exits the loop.

4. Run the load test

import http from 'k6/http';
import { check } from 'k6';
export const options = { vus: 20, duration: '60s' };
export default function () {
  const id = `${__VU}-${__ITER}`;
  const response = http.post(__ENV.FUNCTION_URL, JSON.stringify({ orderId: id, amount: 125 }),
    { headers: { 'Content-Type': 'application/json', 'x-functions-key': __ENV.FUNCTION_KEY } });
  check(response, { accepted: r => r.status === 202 });
}

5. Verify the complete path

Compare the accepted Function request count, queue active-message count, Logic App run history, dead-letter count, and unique Table Storage row count. Increase virtual users until the queue grows and watch it drain after load stops.

Expected result: HTTP calls return 202, Event Grid starts the Logic App, the queue returns to zero active messages with no dead letters, and the Orders table contains one row per unique order ID.

Production extension: Use Premium when you need isolated capacity and predictable latency; use private endpoints, zone redundancy where available, Application Insights, DLQ alerts, and idempotent storage writes.
