Lab 5 - Scale orders from HTTP to storage

Build and test a complete serverless path through Java Functions, Service Bus, Event Grid, Logic Apps, and Table Storage.

90 minutesAzure FunctionsServerless integration

Case study: absorb a checkout surge

Flow: Azure Load Testing calls an HTTP-triggered Java Function. The Function sends an order to orders. When no receiver is active, Service Bus emits an Event Grid notification. A Logic App drains available messages, writes rows to Table Storage, and completes each message.

flowchart LR
  ALT[Azure Load Testing] -- HTTP --> FN[Java Function]
  FN -- managed identity --> SB[orders queue]
  SB -- no-listener event --> EG[Event Grid]
  EG -- trigger --> LA[Logic App]
  LA -- drain and complete --> SB
  LA -- upsert --> TS[Table Storage]

Outcome: Load-test an HTTP endpoint, buffer accepted orders, wake a serverless consumer, and verify that every unique order becomes a table row.

1. Provision the supporting resources

Reuse rg-sb-workshop, the Service Bus namespace, and the orders queue from Lab 1. Do not create another namespace or queue. If orders is missing, return to Lab 1 and create it before continuing.

Option A: Azure portal

  1. In the Azure portal, create a Storage account in rg-sb-workshop. Choose a unique name, the same region as the workshop namespace, Standard performance, and Locally-redundant storage (LRS).
  2. Open the storage account, select Storage browser > Tables > Add table, and create a table named Orders.
  3. Create a Function App in rg-sb-workshop. Select Consumption hosting, the same region, Java 17 on Linux, and the storage account you just created. Enter a globally unique app name such as func-sb-java-<unique>.
  4. Open the Function App, select Settings > Identity, turn the system-assigned identity On, and save.
  5. Open the Lab 1 Service Bus namespace and copy its Host name from the Overview page, for example sb-contoso-12345.servicebus.windows.net. Do not include https://, a trailing slash, or /orders.
  6. Return to the Function App. Under Settings > Environment variables, add SERVICEBUS_FQNS and paste that Host name as its value, then apply the change.
  7. Open the Service Bus namespace and select Access control (IAM) > Add role assignment. Assign Azure Service Bus Data Sender to the Function App's managed identity.

Option B: Azure CLI

mvn archetype:generate "-DarchetypeGroupId=com.microsoft.azure" "-DarchetypeArtifactId=azure-functions-archetype" "-DjavaVersion=17" "-DgroupId=com.contoso.functions" "-DartifactId=order-api" "-DappName=$functionApp" "-DappRegion=eastus" "-DresourceGroup=$resourceGroup" "-DinteractiveMode=false"

# Add azure-messaging-servicebus and azure-identity to pom.xml, then replace Function.java.
mvn clean package
mvn azure-functions:deploy

Add com.azure:azure-messaging-servicebus:7.17.15 and com.azure:azure-identity:1.15.3 to the generated POM. Replace src/main/java/com/contoso/functions/Function.java with:

package com.contoso.functions;
import com.azure.identity.DefaultAzureCredentialBuilder;
import com.azure.messaging.servicebus.*;
import com.microsoft.azure.functions.*;
import com.microsoft.azure.functions.annotation.*;
import java.util.*;

public class Function {
  private static final ServiceBusSenderClient SENDER = new ServiceBusClientBuilder()
    .fullyQualifiedNamespace(System.getenv("SERVICEBUS_FQNS"))
    .credential(new DefaultAzureCredentialBuilder().build())
    .sender().queueName("orders").buildClient();

  @FunctionName("SubmitOrder")
  public HttpResponseMessage run(
      @HttpTrigger(name = "request", methods = HttpMethod.POST,
          authLevel = AuthorizationLevel.FUNCTION) HttpRequestMessage<Optional<String>> request,
      ExecutionContext context) {
    String body = request.getBody().orElseThrow();
    String orderId = request.getHeaders().getOrDefault("x-order-id", UUID.randomUUID().toString());
    SENDER.sendMessage(new ServiceBusMessage(body).setMessageId(orderId)
        .setSubject("OrderCreated").setContentType("application/json"));
    return request.createResponseBuilder(HttpStatus.ACCEPTED).body(orderId).build();
  }
}

3. Configure the cloud consumer

  1. Enable Function managed identity and assign Azure Service Bus Data Sender.
  2. Create a Logic App workflow with a managed-identity Service Bus connection and a Table Storage connection.
  3. Create an Event Grid system topic for the Service Bus namespace and route Microsoft.ServiceBus.ActiveMessagesAvailableWithNoListeners for orders to the Logic App request trigger.
  4. In an Until loop, peek-lock receive a message, parse orderId, insert or replace a row with partition key orders, then complete the message. Stop when receive returns empty.
Important: Event Grid is a wake-up signal, not one event per Service Bus message. The Logic App must drain the entity and make the Table write idempotent.

4. Run the load test

Stop all console receivers. Save this as orders.js, upload it to Azure Load Testing, and configure FUNCTION_URL and FUNCTION_KEY as test parameters.

import http from 'k6/http';
import { check } from 'k6';
export const options = { vus: 20, duration: '60s' };

export default function () {
  const orderId = `${__VU}-${__ITER}`;
  const response = http.post(
    `${__ENV.FUNCTION_URL}?code=${__ENV.FUNCTION_KEY}`,
    JSON.stringify({ orderId, product: 'AZ-BOOK', quantity: 1 }),
    { headers: { 'Content-Type': 'application/json', 'x-order-id': orderId } });
  check(response, { accepted: result => result.status === 202 });
}

For a local smoke test before Azure Load Testing:

5. Verify the complete path

Compare accepted Function requests, queue active messages, Logic App run history, dead-letter count, and unique Table Storage rows. Increase virtual users until the queue grows, then watch it drain after load stops.

Expected result: HTTP calls return 202, Event Grid starts the Logic App, the queue returns to zero active messages with no dead letters, and the Orders table contains one row per unique order ID.

Production extension: Add Application Insights, DLQ alerts, private endpoints, and idempotent storage writes. Choose Premium when isolated capacity and predictable latency are required.
