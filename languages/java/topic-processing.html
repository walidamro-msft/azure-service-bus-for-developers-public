Topics and subscriptions in Java

Publish events, process independent subscription copies, and manage filtering with the Azure Service Bus Java SDK.

Java 17+azure-messaging-servicebusPublisher and processor clients

Package, identity, and topology

Add azure-messaging-servicebus and azure-identity. Reuse clients instead of rebuilding them per message. Grant sender and receiver roles at the narrowest practical scope.

<dependency>
  <groupId>com.azure</groupId>
  <artifactId>azure-messaging-servicebus</artifactId>
  <version>7.x.x</version>
</dependency>
<dependency>
  <groupId>com.azure</groupId>
  <artifactId>azure-identity</artifactId>
  <version>1.x.x</version>
</dependency>
Fan-out boundary: each matching subscription receives an independent durable copy. Two processors on one subscription compete; processors on different subscriptions each receive a copy.

Publish to a topic

Select the topic with sender().topicName(...). Subscription rules inspect system and application properties, not fields hidden in the serialized body.

TokenCredential credential = new DefaultAzureCredentialBuilder().build();
String fqns = System.getenv("SERVICEBUS_FQNS");

try (ServiceBusSenderClient sender = new ServiceBusClientBuilder()
    .fullyQualifiedNamespace(fqns)
    .credential(credential)
    .sender()
    .topicName("order-events")
    .buildClient()) {

  ServiceBusMessage message = new ServiceBusMessage(
      "{\"orderId\":\"ORD-123\",\"total\":149.50}")
      .setMessageId("order-created:ORD-123")
      .setCorrelationId("order:ORD-123")
      .setSubject("OrderCreated.v1")
      .setContentType("application/json");
  message.getApplicationProperties().put("region", "us");
  message.getApplicationProperties().put("priority", 3);
  sender.sendMessage(message);
}

Preserve messageId across retries of the same publish. Each subscriber can then use it as an idempotency key for its own side effects.

Process one subscription

Configure both topicName and subscriptionName. Build a separate processor for each independently scaled capability.

ServiceBusProcessorClient processor = new ServiceBusClientBuilder()
    .fullyQualifiedNamespace(fqns)
    .credential(credential)
    .processor()
    .topicName("order-events")
    .subscriptionName("inventory")
    .disableAutoComplete()
    .maxConcurrentCalls(4)
    .maxAutoLockRenewDuration(Duration.ofMinutes(5))
    .processMessage(context -> {
      try {
        reserveInventory(context.getMessage().getBody().toString());
        context.complete();
      } catch (InvalidOrderException exception) {
        context.deadLetter(new DeadLetterOptions()
            .setDeadLetterReason("InvalidOrder")
            .setDeadLetterErrorDescription(exception.getMessage()));
      } catch (RuntimeException exception) {
        context.abandon();
      }
    })
    .processError(context -> System.err.println(context.getException()))
    .buildProcessorClient();

processor.start();

For pull-based handling, build a receiver with receiver().topicName(...).subscriptionName(...). Settlement changes only the selected subscription copy.

Manage subscription rules

The default $Default rule matches every message. Remove it before adding selective rules. Use SQL filters for expressions and correlation filters for optimized exact matching.

ServiceBusAdministrationClient admin = new ServiceBusAdministrationClientBuilder()
    .fullyQualifiedNamespace(fqns)
    .credential(credential)
    .buildClient();

admin.deleteRule("order-events", "inventory", RuleProperties.DEFAULT_RULE_NAME);
admin.createRule("order-events", "inventory",
    new CreateRuleOptions()
        .setName("UsOrders")
        .setFilter(new SqlRuleFilter("region = 'us' AND priority >= 2")));

admin.createRule("order-events", "billing",
    new CreateRuleOptions()
        .setName("OrderCreated")
        .setFilter(new CorrelationRuleFilter().setSubject("OrderCreated.v1")));

Keep property names, casing, and value types stable. A Java integer property and a string containing digits are different values to the broker.

Operational design

  • Monitor backlog age, active count, and dead-letter count for every subscription.
  • Scale processors independently by subscription and downstream capacity.
  • Keep processor clients alive for the worker lifetime and close them gracefully.
  • Use a dedicated dead-letter receiver for each subscription recovery path.
  • Keep administration clients and rule changes off the message hot path.
