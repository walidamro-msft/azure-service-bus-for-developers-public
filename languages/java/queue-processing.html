Working with queues in Java

Map Service Bus processing decisions to the fluent client builder, processor clients, synchronous and asynchronous receivers, settlement contexts, and sessions.

Java 17+azure-messaging-servicebusProcessor and receiver models

Package, authentication, and builder

Add com.azure:azure-messaging-servicebus and com.azure:azure-identity. Use DefaultAzureCredential locally and managed identity for Azure-hosted workloads. Grant producers Azure Service Bus Data Sender and consumers Azure Service Bus Data Receiver.

<dependency>
  <groupId>com.azure</groupId>
  <artifactId>azure-messaging-servicebus</artifactId>
  <version>7.17.17</version>
</dependency>
<dependency>
  <groupId>com.azure</groupId>
  <artifactId>azure-identity</artifactId>
</dependency>
String fqns = System.getenv("SERVICEBUS_FQNS");
TokenCredential credential = new DefaultAzureCredentialBuilder().build();

ServiceBusClientBuilder builder = new ServiceBusClientBuilder()
    .fullyQualifiedNamespace(fqns)
    .credential(credential);
Connection reuse: clients built from the same top-level ServiceBusClientBuilder can share the underlying connection. Keep clients long-lived and close them during shutdown.

Send to a queue

ServiceBusSenderClient sender = builder.sender()
    .queueName("orders-processing")
    .buildClient();

ServiceBusMessage message = new ServiceBusMessage(BinaryData.fromString(
    "{\"orderId\":\"ORD-123\",\"status\":\"created\"}"))
    .setMessageId(UUID.randomUUID().toString())
    .setCorrelationId("order:ORD-123")
    .setSubject("OrderCreated.v1")
    .setContentType("application/json")
    .setTimeToLive(Duration.ofMinutes(30));
message.getApplicationProperties().put("schemaVersion", 1);

sender.sendMessage(message);

Reuse the sender and close it during shutdown. Use ServiceBusMessageBatch.tryAddMessage for size-aware batching; when it returns false, send the current batch and start another.

Continuous processing with ServiceBusProcessorClient

The processor client is the callback-based production model. It handles background receive and connection recovery. Configure the message and error callbacks before building it.

ServiceBusProcessorClient processor = builder.processor()
    .queueName("orders")
    .receiveMode(ServiceBusReceiveMode.PEEK_LOCK)
    .disableAutoComplete()
    .maxConcurrentCalls(4)
    .maxAutoLockRenewDuration(Duration.ofMinutes(5))
    .prefetchCount(0)
    .identifier("orders-worker:" + System.getenv("COMPUTERNAME"))
    .processMessage(context -> {
        try {
            processOrder(context.getMessage().getBody().toString());
            context.complete();
        } catch (PermanentOrderException exception) {
            context.deadLetter(new DeadLetterOptions()
                .setDeadLetterReason("InvalidOrder")
                .setDeadLetterErrorDescription(exception.getMessage()));
        } catch (RuntimeException exception) {
            context.abandon();
        }
    })
    .processError(context -> System.err.printf("%s %s: %s%n",
        context.getErrorSource(), context.getEntityPath(), context.getException()))
    .buildProcessorClient();

processor.start();
shutdownSignal.await();
processor.stop();
processor.close();

start() returns immediately. On shutdown, stop new delivery and close the processor after in-flight work has had an opportunity to finish.

Processor builder mapping

disableAutoComplete()

Data typeFluent switch
DefaultAuto-complete enabled
Behavior

Without this call, an unsettled message is completed after a successful callback and abandoned when the callback throws.

Strategy

Disable auto-complete when code must explicitly call complete, abandon, defer, or deadLetter.

maxConcurrentCalls(int)

Data typePositive int
Default1
Behavior

Limits simultaneous message callbacks on one processor.

Strategy

Multiply by replica count for potential application concurrency. Increase only while downstream pools, latency, memory, and lock completion remain healthy.

maxAutoLockRenewDuration(Duration)

Data typeDuration
Default5 minutes
Behavior

Sets how long the processor renews a message lock while its callback runs.

Strategy

Use a finite duration above expected processing time. Use Duration.ZERO to disable auto renewal.

prefetchCount(int)

Data typeNon-negative int
DefaultSDK-managed conservative value
Behavior

Sets the number of messages cached locally ahead of callback demand. PeekLock time runs while cached.

Strategy

Use zero while establishing correctness, then increase gradually for fast and steady work. Recheck the API reference for the exact default in the package version you deploy.

receiveMode(ServiceBusReceiveMode)

Data typeEnumeration
DefaultPEEK_LOCK
Behavior

PEEK_LOCK requires settlement; RECEIVE_AND_DELETE removes on transfer.

Strategy

Use PEEK_LOCK unless loss after client transfer is explicitly acceptable.

subQueue(SubQueue)

Data typeEnumeration
DefaultMain entity
Behavior

Selects a dead-letter or transfer dead-letter subqueue instead of the main entity.

Strategy

Build a separate receiver or processor with SubQueue.DEAD_LETTER_QUEUE for controlled recovery.

identifier(String)

Data typeString
DefaultGenerated unique value
Behavior

Names the receive client in selected errors and logs without changing routing.

Strategy

Use a stable role plus unique host or replica suffix.

Pull receiving

Build a synchronous receiver for blocking application-controlled batches, or buildAsyncClient() for a Reactor Flux-based pipeline.

try (ServiceBusReceiverClient receiver = builder.receiver()
        .queueName("orders")
        .receiveMode(ServiceBusReceiveMode.PEEK_LOCK)
        .disableAutoComplete()
        .prefetchCount(0)
        .identifier("orders-batch:" + System.getenv("COMPUTERNAME"))
        .buildClient()) {

    receiver.receiveMessages(20, Duration.ofSeconds(5)).forEach(message -> {
        try {
            processOrder(message.getBody().toString());
            receiver.complete(message);
        } catch (RuntimeException exception) {
            receiver.abandon(message);
        }
    });
}

The batch can contain fewer than the requested number. The async receiver exposes receiveMessages() and asynchronous settlement methods; keep the reactive subscription and receiver alive until shutdown.

Sessions, subscriptions, and dead-letter queues

  • For a topic subscription, configure both topicName and subscriptionName on the processor or receiver sub-builder.
  • For continuous session processing, use sessionProcessor() and set maxConcurrentSessions.
  • For explicit session control, use sessionReceiver(), then acceptSession or acceptNextSession.
  • For dead-letter recovery, set subQueue(SubQueue.DEAD_LETTER_QUEUE) on a separate receiver or processor.
