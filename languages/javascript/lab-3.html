Lab 3 - Preserve each order's event sequence

Provision a session-enabled queue, send interleaved workflows, and process each order in FIFO order.

60 minutesTwo Node.js appsSessions

Case study: ordered fulfilment workflows

Payment, allocation, and dispatch events for many orders may share one queue. Events from different orders can run concurrently, but no order may dispatch before its own payment and allocation steps. A Service Bus session groups each order's messages under its order ID.

flowchart LR
  S[Workflow producer] -- SessionId = order ID --> Q[order-sessions queue]
  Q -- one locked session --> W1[Worker]
  Q -- another locked session --> W2[Worker]

1. Provision a session-enabled queue

Reuse the namespace from Lab 1. Sessions cannot be enabled after a queue is created, so enable session support during creation.

Option A: Azure portal

  1. In the Azure portal, open the Service Bus namespace you created in Lab 1.
  2. Under Entities, select Queues, then select + Queue.
  3. Set the queue name to order-sessions.
  4. Enable Sessions, set Max delivery count to 5, keep the other defaults, and select Create.
  5. On the namespace Overview page, copy the hostname ending in .servicebus.windows.net and set it as SERVICEBUS_FQNS in every terminal used by this lab.
Important: Sessions cannot be enabled after creation. If you create the queue without sessions, delete it and create it again with Sessions enabled.

Option B: Azure CLI

Reuse the PowerShell variables from Lab 1.

az servicebus queue create --resource-group $resourceGroup --namespace-name $namespace --name order-sessions --enable-session true --max-delivery-count 5
$env:SERVICEBUS_FQNS = "$namespace.servicebus.windows.net"
Note: Complete either Option A or Option B, not both.

3. Send two interleaved order workflows

Create workflow-producer/index.js:

import { ServiceBusClient } from "@azure/service-bus";
import { DefaultAzureCredential } from "@azure/identity";

const fqns = process.env.SERVICEBUS_FQNS;
if (!fqns) throw new Error("Set SERVICEBUS_FQNS first.");

const client = new ServiceBusClient(fqns, new DefaultAzureCredential());
const sender = client.createSender("order-sessions");
const orderIds = ["ORD-101", "ORD-202"];
const states = ["PaymentAccepted", "StockAllocated", "Dispatched"];

try {
  for (const state of states) {
    for (const orderId of orderIds) {
      await sender.sendMessages({
        body: state,
        sessionId: orderId,
        messageId: `${orderId}-${state}`,
        subject: state
      });
      console.log(`Sent ${orderId}: ${state}`);
    }
  }
} finally {
  await sender.close();
  await client.close();
}

4. Process sessions concurrently

Create workflow-processor/index.js:

import { ServiceBusClient } from "@azure/service-bus";
import { DefaultAzureCredential } from "@azure/identity";

const fqns = process.env.SERVICEBUS_FQNS;
if (!fqns) throw new Error("Set SERVICEBUS_FQNS first.");

const client = new ServiceBusClient(fqns, new DefaultAzureCredential());
let stopping = false;

async function processNextSession() {
  while (!stopping) {
    try {
      const receiver = await client.acceptNextSession("order-sessions", {
        receiveMode: "peekLock",
        maxAutoLockRenewalDurationInMs: 300000
      });
      try {
        while (!stopping) {
          const messages = await receiver.receiveMessages(10, { maxWaitTimeInMs: 3000 });
          if (messages.length === 0) break;
          for (const message of messages) {
            console.log(`${message.sessionId}: ${message.body}`);
            await new Promise(resolve => setTimeout(resolve, 300));
            await receiver.completeMessage(message);
          }
        }
      } finally {
        await receiver.close();
      }
    } catch (error) {
      if (error.code !== "ServiceTimeout") console.error(error);
    }
  }
}

console.log("Two session workers are running. Press Ctrl+C to stop.");
const workers = [processNextSession(), processNextSession()];
process.on("SIGINT", async () => {
  stopping = true;
  await Promise.allSettled(workers);
  await client.close();
  process.exit(0);
});

5. Run and verify ordering

# Terminal 1
node .\workflow-processor\index.js

# Terminal 2
node .\workflow-producer\index.js

Expected result: Output for ORD-101 and ORD-202 may interleave, but each order always prints PaymentAccepted, StockAllocated, then Dispatched. The queue returns to zero active messages.

Challenge: Add a third order and a third worker. Then run only one worker and compare throughput without changing per-order ordering.
