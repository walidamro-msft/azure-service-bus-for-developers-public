Topics and subscriptions in JavaScript / TypeScript

Publish events, subscribe independently, and manage broker-side rules with @azure/service-bus.

Node.js 20+@azure/service-busAsync publisher and subscribers

Package, identity, and topology

Install the Service Bus and identity packages. Reuse one ServiceBusClient and long-lived entity clients. Use sender, receiver, and owner roles according to the operation.

npm install @azure/service-bus @azure/identity
import {
  ServiceBusAdministrationClient,
  ServiceBusClient
} from "@azure/service-bus";
import { DefaultAzureCredential } from "@azure/identity";

const fqns = process.env.SERVICEBUS_FQNS;
if (!fqns) throw new Error("Set SERVICEBUS_FQNS.");
const credential = new DefaultAzureCredential();
const client = new ServiceBusClient(fqns, credential);
Fan-out boundary: different subscriptions receive independent copies. Multiple receivers attached to the same subscription compete for that subscription's messages.

Publish to a topic

Create a sender for the topic. Put filterable metadata in applicationProperties; the broker does not deserialize the body for subscription rules.

const sender = client.createSender("order-events");
try {
  await sender.sendMessages({
    body: { orderId: "ORD-123", total: 149.50 },
    messageId: "order-created:ORD-123",
    correlationId: "order:ORD-123",
    subject: "OrderCreated.v1",
    contentType: "application/json",
    applicationProperties: { region: "us", priority: 3 }
  });
} finally {
  await sender.close();
}

Reuse the same messageId for retries of one logical event. Use a new ID for a later event about the same aggregate.

Process one subscription

Create a receiver with topic and subscription names. A receiver subscription is callback-driven; receiveMessages provides explicit pull control.

const receiver = client.createReceiver("order-events", "inventory", {
  receiveMode: "peekLock",
  maxAutoLockRenewalDurationInMs: 300_000
});

const handlers = receiver.subscribe({
  processMessage: async message => {
    try {
      await reserveInventory(message.body);
      await receiver.completeMessage(message);
    } catch (error) {
      if (error instanceof InvalidOrderError) {
        await receiver.deadLetterMessage(message, {
          deadLetterReason: "InvalidOrder",
          deadLetterErrorDescription: error.message
        });
      } else {
        await receiver.abandonMessage(message);
      }
    }
  },
  processError: async args => console.error(args.error)
}, {
  autoCompleteMessages: false,
  maxConcurrentCalls: 4
});

Close the subscription, receiver, and shared client during graceful shutdown. Settlement affects only the inventory copy.

Manage subscription rules

A subscription starts with $Default, which matches all messages. Delete it before creating selective rules.

const admin = new ServiceBusAdministrationClient(fqns, credential);

await admin.deleteRule("order-events", "inventory", "$Default");
await admin.createRule("order-events", "inventory", "UsOrders", {
  sqlExpression: "region = 'us' AND priority >= 2"
});
await admin.createRule("order-events", "billing", "OrderCreated", {
  subject: "OrderCreated.v1"
});

Operational design

  • Deploy one independently scalable worker role per subscription capability.
  • Monitor active, dead-letter, and oldest-message age per subscription.
  • Keep receivers long-lived; do not create one for every message.
  • Create a dedicated receiver with subQueueType: "deadLetter" for recovery.
  • Keep administration operations outside the publish and receive hot paths.
