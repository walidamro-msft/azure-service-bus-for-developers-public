Lab 4 - Diagnose and recover poison messages

Create an invalid order, dead-letter it with diagnostics, repair it, and re-drive it safely.

60 minutesThree Node.js appsDead-letter queue

Case study: isolate an invalid order

A malformed order must not block valid work or disappear without evidence. The validator moves it to the queue's dead-letter subqueue with a reason and description. A recovery operator inspects that metadata, creates a corrected copy, and completes the original only after the replacement is accepted.

flowchart LR
  P[Poison producer] -- invalid order --> Q[orders queue]
  Q -- validation failure --> D[orders DLQ]
  D -- inspect and repair --> R[Recovery app]
  R -- corrected copy --> Q
Important: Service Bus does not automatically clean the DLQ. Production systems should alert on dead-letter counts and retain reason, description, and correlation identifiers.

2. Send an invalid order

Create poison-producer/index.js:

import { ServiceBusClient } from "@azure/service-bus";
import { DefaultAzureCredential } from "@azure/identity";

const fqns = process.env.SERVICEBUS_FQNS;
if (!fqns) throw new Error("Set SERVICEBUS_FQNS first.");

const client = new ServiceBusClient(fqns, new DefaultAzureCredential());
const sender = client.createSender("orders");

try {
  await sender.sendMessages({
    body: "not-json",
    messageId: `POISON-${Date.now()}`,
    subject: "OrderCreated"
  });
  console.log("Sent one invalid order.");
} finally {
  await sender.close();
  await client.close();
}

3. Validate and dead-letter

Create order-validator/index.js:

import { ServiceBusClient } from "@azure/service-bus";
import { DefaultAzureCredential } from "@azure/identity";

const fqns = process.env.SERVICEBUS_FQNS;
if (!fqns) throw new Error("Set SERVICEBUS_FQNS first.");

const client = new ServiceBusClient(fqns, new DefaultAzureCredential());
const receiver = client.createReceiver("orders", { receiveMode: "peekLock" });

try {
  const [message] = await receiver.receiveMessages(1, { maxWaitTimeInMs: 20000 });
  if (!message) throw new Error("No order was available.");

  if (typeof message.body !== "object" || message.body === null || !message.body.orderId) {
    await receiver.deadLetterMessage(message, {
      deadLetterReason: "InvalidOrderJson",
      deadLetterErrorDescription: "The body must be a JSON object with an orderId."
    });
    console.log(`Dead-lettered ${message.messageId}: InvalidOrderJson`);
  } else {
    await receiver.completeMessage(message);
    console.log(`Validated and completed ${message.body.orderId}.`);
  }
} finally {
  await receiver.close();
  await client.close();
}

4. Inspect, repair, and re-drive

Create dlq-recovery/index.js:

import { ServiceBusClient } from "@azure/service-bus";
import { DefaultAzureCredential } from "@azure/identity";

const fqns = process.env.SERVICEBUS_FQNS;
if (!fqns) throw new Error("Set SERVICEBUS_FQNS first.");

const client = new ServiceBusClient(fqns, new DefaultAzureCredential());
const dlq = client.createReceiver("orders", {
  subQueueType: "deadLetter",
  receiveMode: "peekLock"
});
const sender = client.createSender("orders");

try {
  const [failed] = await dlq.receiveMessages(1, { maxWaitTimeInMs: 20000 });
  if (!failed) throw new Error("The DLQ is empty.");

  console.log(`Reason: ${failed.deadLetterReason}`);
  console.log(`Description: ${failed.deadLetterErrorDescription}`);
  const repaired = {
    body: { orderId: "ORD-RECOVERED", product: "AZ-BOOK", quantity: 1 },
    messageId: `${failed.messageId}-repaired`,
    correlationId: String(failed.messageId),
    subject: failed.subject,
    contentType: "application/json"
  };

  await sender.sendMessages(repaired);
  await dlq.completeMessage(failed);
  console.log(`Re-drove ${repaired.messageId} and completed the DLQ original.`);
} finally {
  await sender.close();
  await dlq.close();
  await client.close();
}
Recovery rule: Complete the original only after the repaired copy is accepted. Preserve correlation and use idempotent storage writes.

5. Run the recovery drill

node .\poison-producer\index.js
node .\order-validator\index.js
# Verify the portal shows one dead-lettered message.
node .\dlq-recovery\index.js
# Run the validator again to consume the repaired copy.
node .\order-validator\index.js

Expected result: The DLQ count rises to one after validation, then returns to zero after recovery. The final validator run completes ORD-RECOVERED, and the active queue count returns to zero.

Challenge: Change the recovery policy so unknown dead-letter reasons are abandoned for investigation rather than automatically re-driven.
