Lab 2 - Publish one event to two subscribers

Create a topic, build a Node.js publisher, and run one reusable subscriber for two independent subscriptions.

60 minutesThree processesTopics

Case study: notify inventory and customer care

When checkout confirms an order, inventory must reserve stock and customer care must prepare a confirmation. The publisher sends one OrderCreated event. Service Bus copies it into two durable subscriptions so both business capabilities receive the event independently.

flowchart LR
  P[Order publisher] -- one message --> T[order-events topic]
  T -- copy --> I[inventory subscription]
  T -- copy --> C[customer-care subscription]
  I -- receive --> SI[Subscriber instance: inventory]
  C -- receive --> SC[Subscriber instance: customer-care]
Important: Run each receiver against a different subscription. Two receivers using the same subscription are competing consumers and would split messages instead of both receiving a copy.

1. Provision the topic and subscriptions

Reuse the namespace from Lab 1.

Option A: Azure portal

  1. In the Azure portal, open the Service Bus namespace you created in Lab 1.
  2. Under Entities, select Topics, then select + Topic.
  3. Set the topic name to order-events, keep the other defaults, and select Create.
  4. Open the order-events topic, select Subscriptions, then select + Subscription.
  5. Set the subscription name to inventory, keep the other defaults, and select Create.
  6. Select + Subscription again, create a subscription named customer-care, and keep the other defaults.
  7. On the namespace Overview page, copy the hostname ending in .servicebus.windows.net and set it as SERVICEBUS_FQNS in every terminal used by this lab.

Option B: Azure CLI

Reuse the PowerShell variables from Lab 1.

az servicebus topic create --resource-group $resourceGroup --namespace-name $namespace --name order-events
az servicebus topic subscription create --resource-group $resourceGroup --namespace-name $namespace --topic-name order-events --name inventory
az servicebus topic subscription create --resource-group $resourceGroup --namespace-name $namespace --topic-name order-events --name customer-care
$env:SERVICEBUS_FQNS = "$namespace.servicebus.windows.net"
Note: Complete either Option A or Option B, not both.

Checkpoint: The portal shows one topic and two subscriptions, each with an active message count of zero.

2. Create publisher and subscriber applications

mkdir lab2-topic
cd lab2-topic
mkdir orders-publisher, events-subscriber

foreach ($app in @("orders-publisher", "events-subscriber")) {
  Push-Location $app
  npm init -y
  npm pkg set type=module
  npm install @azure/service-bus @azure/identity
  Pop-Location
}

The subscriber accepts a subscription name as a command-line argument, so the same application can run as two independent business consumers.

3. Build the order publisher

Create orders-publisher/index.js:

import { ServiceBusClient } from "@azure/service-bus";
import { DefaultAzureCredential } from "@azure/identity";

const fqns = process.env.SERVICEBUS_FQNS;
if (!fqns) throw new Error("Set SERVICEBUS_FQNS first.");

const client = new ServiceBusClient(fqns, new DefaultAzureCredential());
const sender = client.createSender("order-events");
const orderId = `ORD-${new Date().toISOString().replaceAll(/\D/g, "").slice(8, 14)}`;

try {
  await sender.sendMessages({
    body: { orderId, product: "AZ-BOOK", quantity: 2 },
    messageId: orderId,
    subject: "OrderCreated",
    contentType: "application/json"
  });
  console.log(`Published one event for ${orderId}`);
} finally {
  await sender.close();
  await client.close();
}

4. Build the reusable subscriber

Create events-subscriber/index.js:

import { ServiceBusClient } from "@azure/service-bus";
import { DefaultAzureCredential } from "@azure/identity";

const subscriptionName = process.argv[2];
if (!["inventory", "customer-care"].includes(subscriptionName)) {
  throw new Error("Pass inventory or customer-care.");
}

const fqns = process.env.SERVICEBUS_FQNS;
if (!fqns) throw new Error("Set SERVICEBUS_FQNS first.");

const client = new ServiceBusClient(fqns, new DefaultAzureCredential());
const receiver = client.createReceiver("order-events", subscriptionName, { receiveMode: "peekLock" });

receiver.subscribe({
  processMessage: async message => {
    console.log(`[${subscriptionName}] received ${message.messageId}:`, message.body);
    await receiver.completeMessage(message);
  },
  processError: async args => console.error(`[${subscriptionName}]`, args.error)
}, { autoCompleteMessages: false });

console.log(`Listening on ${subscriptionName}. Press Ctrl+C to stop.`);
process.on("SIGINT", async () => {
  await receiver.close();
  await client.close();
  process.exit(0);
});

5. Run three processes and verify fan-out

# Terminal 1
node .\events-subscriber\index.js inventory

# Terminal 2
node .\events-subscriber\index.js customer-care

# Terminal 3
node .\orders-publisher\index.js

Expected result: The publisher reports one sent event. Both subscriber terminals display the same MessageId and body. Each subscription's active message count returns to zero.

Challenge: Stop only the customer-care process and publish another event. Inventory processes it immediately while the customer-care subscription retains its own copy. Restart customer-care and observe it receive the stored event.
