Lab 5 - Scale orders from HTTP to storage

Build and test a complete Node.js serverless path through Functions, Service Bus, Event Grid, Logic Apps, and Table Storage.

90 minutesNode.js 20 FunctionServerless integration

Case study: absorb a checkout surge

Flow: Azure Load Testing calls an HTTP-triggered Node.js Function. The Function sends an order to orders. When no receiver is active, Service Bus emits an Event Grid notification. A Logic App handles that notification, drains available messages with peek-lock, writes rows to Azure Table Storage, and completes each message.

flowchart LR
  ALT[Azure Load Testing] -- HTTP POST --> FN[Node.js Function producer]
  FN -- managed identity --> SB[Service Bus orders queue]
  SB -- queue has messages event --> EG[Event Grid]
  EG -- trigger --> LA[Logic App consumer]
  LA -- receive and complete --> SB
  LA -- upsert order --> TS[Table Storage]

Outcome: Load-test an HTTP endpoint, buffer accepted orders, wake a serverless consumer, and verify that every unique order becomes a table row.

1. Provision the supporting resources

Reuse rg-sb-workshop, the Service Bus namespace, and the orders queue from Lab 1. Do not create another namespace or queue. If orders is missing, return to Lab 1 and create it before continuing. The Function uses Flex Consumption, Node.js 20, Functions runtime v4, and Application Insights.

Option A: Azure portal

  1. In the Azure portal, create a Storage account in rg-sb-workshop. Choose a unique name, the same region as the workshop namespace, Standard performance, and Locally-redundant storage (LRS).
  2. Open the storage account, select Storage browser > Tables > Add table, and create a table named Orders.
  3. Create a Function App in rg-sb-workshop. Select Flex Consumption, the same region, Node.js 20, and the storage account you just created. Enter a globally unique app name such as func-sb-<unique>.
  4. Open the Function App, select Settings > Identity, turn the system-assigned identity On, and save.
  5. Open the Lab 1 Service Bus namespace and copy its Host name from the Overview page, for example sb-contoso-12345.servicebus.windows.net. Do not include https://, a trailing slash, or /orders.
  6. Return to the Function App. Under Settings > Environment variables, add SERVICEBUS_FQNS and paste that Host name as its value, then apply the change.
  7. Open the Service Bus namespace and select Access control (IAM) > Add role assignment. Assign Azure Service Bus Data Sender to the Function App's managed identity.
  8. Create a Logic App (Consumption) in rg-sb-workshop, using a unique name such as logic-sb-orders-<unique>. Open Identity, enable its system-assigned identity, and save.
  9. On the Service Bus namespace, assign Azure Service Bus Data Receiver to the Logic App's managed identity. On the storage account, assign Storage Table Data Contributor to the same identity.

Note: Role assignments can take several minutes to propagate. Service Bus access is passwordless: local tools use DefaultAzureCredential, and both cloud components use managed identity.

2. Create, test, and deploy the Function producer

mkdir lab5-function
cd lab5-function
func init . --worker-runtime node --language javascript --model V4
func new --name SubmitOrder --template "HTTP trigger" --authlevel function
npm install @azure/service-bus @azure/identity

Replace src/functions/SubmitOrder.js with:

const { app } = require("@azure/functions");
const { ServiceBusClient } = require("@azure/service-bus");
const { DefaultAzureCredential } = require("@azure/identity");

const fqns = process.env.SERVICEBUS_FQNS;
if (!fqns) throw new Error("Set SERVICEBUS_FQNS first.");

const client = new ServiceBusClient(fqns, new DefaultAzureCredential());
const sender = client.createSender("orders");

app.http("SubmitOrder", {
  methods: ["POST"],
  authLevel: "function",
  handler: async (request, context) => {
    const order = await request.json();
    if (!order?.orderId) {
      return { status: 400, jsonBody: { error: "orderId is required" } };
    }

    await sender.sendMessages({
      body: order,
      messageId: String(order.orderId),
      subject: "OrderCreated",
      contentType: "application/json"
    });
    context.log(`Accepted ${order.orderId}`);
    return { status: 202, jsonBody: { orderId: order.orderId } };
  }
});

3. Configure Event Grid and the Logic App consumer

  1. Open the Logic App designer and choose the When a resource event occurs Event Grid trigger. Select the current subscription, rg-sb-workshop, the Service Bus namespace resource, and event type Microsoft.ServiceBus.ActiveMessagesAvailableWithNoListeners.
  2. Add a trigger condition so only the queue is handled: @endsWith(triggerBody()?['subject'], '/queues/orders'). Saving the workflow creates the Event Grid subscription on the namespace.
  3. Add an Until loop with a Boolean variable named queueEmpty, initially false. Set the loop condition to @equals(variables('queueEmpty'), true).
  4. Inside the loop, use the Service Bus connector with managed identity to Receive messages from a queue (peek-lock). Select orders, request one message, and set a short timeout.
  5. If no message is returned, set queueEmpty to true. Otherwise parse the JSON body, then use the Azure Table Storage connector with managed identity to insert or replace an entity in Orders. Set PartitionKey to orders, RowKey to orderId, and include the message body fields.
  6. After the table write succeeds, use the Service Bus connector to Complete the message with the lock token from the receive action. Configure failure handling so a failed table write does not complete the message.
Important: Event Grid is a wake-up signal, not one event per message, and it does not contain the message body. Stop every console receiver before testing so Service Bus emits the no-listener event; the Logic App must drain the queue until receive returns no message.

4. Run the load test

$hostName = az functionapp show --resource-group $resourceGroup --name $functionApp --query defaultHostName -o tsv
$functionKey = az functionapp function keys list --resource-group $resourceGroup --name $functionApp --function-name SubmitOrder --query default -o tsv
$env:FUNCTION_URL = "https://$hostName/api/SubmitOrder"
$env:FUNCTION_KEY = $functionKey

# Cloud smoke test
$body = @{ orderId = "CLOUD-001"; product = "AZ-BOOK"; quantity = 1 } | ConvertTo-Json
Invoke-RestMethod -Method Post -Uri "$env:FUNCTION_URL?code=$env:FUNCTION_KEY" -ContentType application/json -Body $body

Save this k6 script as load-test.js and upload it to an Azure Load Testing test. Add FUNCTION_URL as an environment variable and FUNCTION_KEY as a secret.

import http from "k6/http";
import { check } from "k6";

export const options = { vus: 20, duration: "60s" };

export default function () {
  const orderId = `LOAD-${__VU}-${__ITER}`;
  const response = http.post(
    `${__ENV.FUNCTION_URL}?code=${__ENV.FUNCTION_KEY}`,
    JSON.stringify({ orderId, product: "AZ-BOOK", quantity: 1 }),
    { headers: { "Content-Type": "application/json" } }
  );
  check(response, { accepted: result => result.status === 202 });
}

5. Verify the complete path

Compare the accepted Function request count in Application Insights, queue active-message count, Logic App run history, dead-letter count, and unique Table Storage row count. Increase virtual users until the queue grows and watch it drain after load stops.

Expected result: Local and cloud HTTP calls return 202. During the load test, Event Grid starts the Logic App, the queue grows and then returns to zero active messages with no dead letters, and the Orders table contains one row per unique order ID.

Production extension: Use Premium when you need isolated Service Bus capacity and predictable latency; add private endpoints, zone redundancy where available, DLQ alerts, and idempotent storage writes.
