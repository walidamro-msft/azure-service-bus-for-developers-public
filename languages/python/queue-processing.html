Working with queues in Python

Map Service Bus processing decisions to receiver iteration, batched receive, explicit settlement, AutoLockRenewer, sessions, and synchronous or asynchronous clients.

Python 3.10+azure-servicebusReceiver loop and batch models

Package, authentication, and client

Install azure-servicebus and azure-identity. Use DefaultAzureCredential locally and managed identity for Azure-hosted workloads. Grant producers Azure Service Bus Data Sender and consumers Azure Service Bus Data Receiver.

python -m pip install azure-servicebus azure-identity
import os

from azure.identity import DefaultAzureCredential
from azure.servicebus import ServiceBusClient

fully_qualified_namespace = os.environ["SERVICEBUS_FQNS"]
credential = DefaultAzureCredential()
client = ServiceBusClient(fully_qualified_namespace, credential)
Production identity: DefaultAzureCredential can select a workload or managed identity in Azure. Do not store a connection string or access key in source code.

Send to a queue

from datetime import timedelta
from uuid import uuid4

from azure.servicebus import ServiceBusMessage

with client:
    with client.get_queue_sender("orders-processing") as sender:
        message = ServiceBusMessage(
            '{"orderId":"ORD-123","status":"created"}',
            message_id=str(uuid4()),
            correlation_id="order:ORD-123",
            subject="OrderCreated.v1",
            content_type="application/json",
            application_properties={"schemaVersion": 1},
            time_to_live=timedelta(minutes=30),
        )
        sender.send_messages(message)

Reuse a sender context for multiple sends. Pass a list to send_messages only when the encoded batch fits, or use create_message_batch and add_message for size-aware batching.

Continuous processing with receiver iteration

The Python SDK does not expose a processor-options object or callback processor. Iterate a ServiceBusReceiver for continuous receive and use AutoLockRenewer when work can exceed the entity lock duration.

from azure.servicebus import AutoLockRenewer, ServiceBusReceiveMode

lock_renewer = AutoLockRenewer(max_lock_renewal_duration=300)

try:
    with client:
        with client.get_queue_receiver(
            queue_name="orders",
            receive_mode=ServiceBusReceiveMode.PEEK_LOCK,
            auto_lock_renewer=lock_renewer,
            prefetch_count=0,
            client_identifier=f"orders-worker:{os.environ.get('COMPUTERNAME', 'local')}",
        ) as receiver:
            for message in receiver:
                try:
                    process_order(str(message))
                    receiver.complete_message(message)
                except PermanentOrderError as error:
                    receiver.dead_letter_message(
                        message,
                        reason="InvalidOrder",
                        error_description=str(error),
                    )
                except Exception:
                    receiver.abandon_message(message)
finally:
    lock_renewer.close()

Settlement is explicit: a successful loop iteration does not automatically complete a message. For cooperative cancellation, break the loop and let the receiver and client context managers close their links and connection.

Receiver and lock-renewer mapping

Settlement policy

APIReceiver settlement methods
DefaultExplicit
Behavior

Call complete_message, abandon_message, defer_message, or dead_letter_message on the receiver.

Strategy

Complete only after durable work succeeds. There is no Python auto_complete_messages receiver option.

Handler concurrency

APIApplication-managed
DefaultSequential loop
Behavior

The synchronous receiver loop processes one iteration at a time unless your code dispatches work. The SDK has no max_concurrent_calls receiver argument.

Strategy

Prefer azure.servicebus.aio with bounded asyncio tasks for I/O-bound concurrency. Preserve the receiver context until every task settles its message.

auto_lock_renewer

Data typeAutoLockRenewer | None
DefaultNone
Behavior

Automatically registers received messages, or the session for a session receiver, with a background lock renewer.

Strategy

Provide one when processing may exceed the lock duration. Size its worker pool for the number of messages or sessions renewed concurrently.

max_lock_renewal_duration

Configure onAutoLockRenewer
Default300 seconds
Behavior

Sets the total automatic renewal period for registered messages or sessions.

Strategy

Use a finite duration above expected successful processing time and close the renewer during shutdown.

prefetch_count

Data typeint
Default0
Behavior

Caches up to this many messages locally. In PeekLock mode, lock time runs while cached.

Strategy

Increase for fast, steady work only. With ReceiveAndDelete, prefetched messages can be lost if the process exits before application code receives them.

receive_mode

Data typeServiceBusReceiveMode
DefaultPEEK_LOCK
Behavior

Accepts PEEK_LOCK or RECEIVE_AND_DELETE.

Strategy

Use PEEK_LOCK for business work. Use RECEIVE_AND_DELETE only when loss after transfer is acceptable.

sub_queue

Data typeServiceBusSubQueue | str | None
DefaultNone
Behavior

Selects the main entity, dead-letter subqueue, or transfer dead-letter subqueue.

Strategy

Create a separate receiver with ServiceBusSubQueue.DEAD_LETTER for deliberate recovery.

client_identifier

Data typestr | None
DefaultGenerated unique value
Behavior

Names the receiver in selected errors without affecting routing or message identity.

Strategy

Use a role plus unique host or replica suffix.

Batched receiving

Use receive_messages for scheduled jobs, bounded processing cycles, or explicit batch control.

with client:
    with client.get_queue_receiver(
        queue_name="orders",
        receive_mode=ServiceBusReceiveMode.PEEK_LOCK,
        prefetch_count=0,
        client_identifier="orders-batch",
    ) as receiver:
        messages = receiver.receive_messages(
            max_message_count=20,
            max_wait_time=5,
        )
        for message in messages:
            try:
                process_order(str(message))
                receiver.complete_message(message)
            except Exception:
                receiver.abandon_message(message)

The method returns as soon as messages arrive and a gap occurs, so the list can contain fewer than max_message_count. The receiver also supports peek_messages, receive_deferred_messages, and renew_message_lock.

Sessions, subscriptions, and dead-letter queues

  • For a topic subscription, call get_subscription_receiver(topic_name, subscription_name, ...).
  • For one session, pass a specific session_id; to accept the next available session, pass NEXT_AVAILABLE_SESSION.
  • When a session receiver gets an AutoLockRenewer, the renewer applies to the session lock rather than each message lock.
  • For dead-letter recovery, pass sub_queue=ServiceBusSubQueue.DEAD_LETTER to a separate receiver.
