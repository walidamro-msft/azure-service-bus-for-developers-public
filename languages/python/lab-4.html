Lab 4 - Diagnose and recover poison messages

Create an invalid order, dead-letter it with diagnostics, repair it, and re-drive it safely.

60 minutesThree Python appsDead-letter queue

Case study: isolate an invalid order

A malformed order must not block valid work or disappear without evidence. The validator moves it to the queue's dead-letter subqueue with a reason and description. A recovery operator inspects that metadata, creates a corrected copy, and completes the original only after the replacement is accepted.

flowchart LR
      P[Poison producer] -- invalid order --> Q[orders queue]
      Q -- validation failure --> D[orders DLQ]
      D -- inspect and repair --> R[Recovery app]
      R -- corrected copy --> Q
Important: Service Bus does not automatically clean the DLQ. Production systems should alert on dead-letter counts and retain reason, description, and correlation identifiers.

2. Send an invalid order

Replace poison_producer.py with:

import os
    from datetime import datetime, timezone

    from azure.identity import DefaultAzureCredential
    from azure.servicebus import ServiceBusClient, ServiceBusMessage

    fully_qualified_namespace = os.environ["SERVICEBUS_FQNS"]
    message_id = f"POISON-{datetime.now(timezone.utc):%H%M%S}"

    with ServiceBusClient(fully_qualified_namespace, DefaultAzureCredential()) as client:
      with client.get_queue_sender(queue_name="orders") as sender:
        sender.send_messages(ServiceBusMessage(
          "not-json",
          message_id=message_id,
          subject="OrderCreated",
        ))

    print(f"Sent one invalid order: {message_id}")

3. Validate and dead-letter

Replace validator.py with:

import json
    import os

    from azure.identity import DefaultAzureCredential
    from azure.servicebus import ServiceBusClient

    fully_qualified_namespace = os.environ["SERVICEBUS_FQNS"]

    with ServiceBusClient(fully_qualified_namespace, DefaultAzureCredential()) as client:
      with client.get_queue_receiver(queue_name="orders", max_wait_time=20) as receiver:
        messages = receiver.receive_messages(max_message_count=1, max_wait_time=20)
        if not messages:
          raise RuntimeError("No order was available.")

        message = messages[0]
        try:
          order = json.loads(str(message))
          order_id = order["orderId"]
          receiver.complete_message(message)
          print(f"Validated and completed {order_id}.")
        except (json.JSONDecodeError, KeyError) as error:
          receiver.dead_letter_message(
            message,
            reason="InvalidOrderJson",
            error_description=str(error),
          )
          print(f"Dead-lettered {message.message_id}: InvalidOrderJson")

4. Inspect, repair, and re-drive

Replace recovery.py with:

import json
    import os

    from azure.identity import DefaultAzureCredential
    from azure.servicebus import ServiceBusClient, ServiceBusMessage, ServiceBusSubQueue

    fully_qualified_namespace = os.environ["SERVICEBUS_FQNS"]

    with ServiceBusClient(fully_qualified_namespace, DefaultAzureCredential()) as client:
      with client.get_queue_receiver(
        queue_name="orders",
        sub_queue=ServiceBusSubQueue.DEAD_LETTER,
        max_wait_time=20,
      ) as dlq, client.get_queue_sender(queue_name="orders") as sender:
        messages = dlq.receive_messages(max_message_count=1, max_wait_time=20)
        if not messages:
          raise RuntimeError("The DLQ is empty.")

        failed = messages[0]
        print(f"Reason: {failed.dead_letter_reason}")
        print(f"Description: {failed.dead_letter_error_description}")

        repaired_body = json.dumps({
          "orderId": "ORD-RECOVERED",
          "product": "AZ-BOOK",
          "quantity": 1,
        })
        repaired = ServiceBusMessage(
          repaired_body,
          message_id=f"{failed.message_id}-repaired",
          correlation_id=str(failed.message_id),
          subject=failed.subject,
          content_type="application/json",
        )
        sender.send_messages(repaired)
        dlq.complete_message(failed)
        print(f"Re-drove {repaired.message_id} and completed the DLQ original.")
Recovery rule: Complete the original only after the repaired copy is accepted. Preserve correlation identifiers and make downstream storage writes idempotent.

Expected result: The DLQ count rises to one after validation, then returns to zero after recovery. The final validator run completes ORD-RECOVERED, and the active queue count returns to zero.

Challenge: Change the recovery policy so unknown dead-letter reasons are abandoned for investigation rather than automatically re-driven.
