Topics and subscriptions in Python

Publish events, receive independent subscription copies, and manage SQL and correlation rules with azure-servicebus.

Python 3.10+azure-servicebusSync and async clients

Package, identity, and topology

Install azure-servicebus and azure-identity. Reuse the namespace client and entity clients. Grant publisher and subscriber identities only the data-plane role each requires.

python -m pip install azure-servicebus azure-identity
import os
from azure.identity import DefaultAzureCredential
from azure.servicebus import ServiceBusClient, ServiceBusMessage

fully_qualified_namespace = os.environ["SERVICEBUS_FQNS"]
credential = DefaultAzureCredential()
client = ServiceBusClient(fully_qualified_namespace, credential)
Fan-out boundary: every matching subscription stores its own durable copy. Receivers on one subscription compete; receivers on different subscriptions do not.

Publish to a topic

Use get_topic_sender. Put routing values in application_properties because subscription rules cannot inspect JSON fields in the body.

import json

body = json.dumps({"orderId": "ORD-123", "total": 149.50})
message = ServiceBusMessage(
    body,
    message_id="order-created:ORD-123",
    correlation_id="order:ORD-123",
    subject="OrderCreated.v1",
    content_type="application/json",
    application_properties={"region": "us", "priority": 3},
)

with client:
    with client.get_topic_sender(topic_name="order-events") as sender:
        sender.send_messages(message)

Reuse message_id for retries of the same logical publish. Each subscription retains that ID and can apply its own idempotency policy.

Process one subscription

Create a receiver with both names. Use a separate long-running receiver loop for every independently deployed subscriber capability.

with ServiceBusClient(fully_qualified_namespace, credential) as client:
    with client.get_subscription_receiver(
        topic_name="order-events",
        subscription_name="inventory",
        max_wait_time=20,
        max_lock_renewal_duration=300,
    ) as receiver:
        for message in receiver:
            try:
                reserve_inventory(str(message))
                receiver.complete_message(message)
            except InvalidOrderError as error:
                receiver.dead_letter_message(
                    message,
                    reason="InvalidOrder",
                    error_description=str(error),
                )
            except Exception:
                receiver.abandon_message(message)

For higher throughput, receive batches and bound application-managed concurrency. Settlement changes only the selected subscription copy.

Manage subscription rules

Use the management client for topology and rule changes. Remove $Default when the subscription should not receive every event.

from azure.servicebus.management import ServiceBusAdministrationClient
from azure.servicebus.management import CorrelationRuleFilter, SqlRuleFilter

admin = ServiceBusAdministrationClient(fully_qualified_namespace, credential)
admin.delete_rule("order-events", "inventory", "$Default")
admin.create_rule(
    "order-events",
    "inventory",
    "UsOrders",
    filter=SqlRuleFilter("region = 'us' AND priority >= 2"),
)
admin.create_rule(
    "order-events",
    "billing",
    "OrderCreated",
    filter=CorrelationRuleFilter(subject="OrderCreated.v1"),
)

Use SQL filters for expressions and correlation filters for exact values. Treat property names, casing, and Python scalar types as part of the publisher contract.

Operational design

  • Monitor active, dead-letter, and oldest-message age for every subscription.
  • Scale subscriber processes independently according to downstream capacity.
  • Keep clients and receivers open for the consuming workflow, then close them gracefully.
  • Use sub_queue=ServiceBusSubQueue.DEAD_LETTER for a dedicated recovery receiver.
  • For async applications, use the equivalent clients from azure.servicebus.aio.
