Lab 3 - Preserve each order's event sequence

Provision a session-enabled queue, send interleaved workflows, and process each order in FIFO order.

60 minutesTwo Python appsSessions

Case study: ordered fulfilment workflows

Payment, allocation, and dispatch events for many orders may share one queue. Events from different orders can run concurrently, but no order may dispatch before its own payment and allocation steps. A Service Bus session groups each order's messages under its order ID.

flowchart LR
  S[Workflow producer] -- SessionId = order ID --> Q[order-sessions queue]
  Q -- one locked session --> W1[Worker]
  Q -- another locked session --> W2[Worker]

1. Provision a session-enabled queue

Reuse the namespace from Lab 1. Session support must be enabled when the queue is created and cannot be enabled after creation.

Option A: Azure portal

  1. In the Azure portal, open the Service Bus namespace you created in Lab 1.
  2. Under Entities, select Queues, then select + Queue.
  3. Set the queue name to order-sessions, enable Sessions, and set Max delivery count to 5.
  4. Keep the other defaults and select Create. Confirm the queue shows sessions as enabled before continuing.
Important: Sessions cannot be enabled after a queue is created. If you create the queue without sessions, delete it and create it again with Sessions enabled.

Option B: Azure CLI

az servicebus queue create --resource-group $resourceGroup --namespace-name $namespace --name order-sessions --enable-session true --max-delivery-count 5
$env:SERVICEBUS_FQNS = "$namespace.servicebus.windows.net"
Note: Complete either Option A or Option B, not both. Reuse the SERVICEBUS_FQNS value from Lab 1.

3. Send two interleaved order workflows

Replace producer.py with:

import os

from azure.identity import DefaultAzureCredential
from azure.servicebus import ServiceBusClient, ServiceBusMessage

fully_qualified_namespace = os.environ["SERVICEBUS_FQNS"]
order_ids = ["ORD-101", "ORD-202"]
states = ["PaymentAccepted", "StockAllocated", "Dispatched"]

with ServiceBusClient(fully_qualified_namespace, DefaultAzureCredential()) as client:
    with client.get_queue_sender(queue_name="order-sessions") as sender:
        for state in states:
            for order_id in order_ids:
                sender.send_messages(ServiceBusMessage(
                    state,
                    session_id=order_id,
                    message_id=f"{order_id}-{state}",
                    subject=state,
                ))
                print(f"Sent {order_id}: {state}")

4. Process one available session at a time

Replace processor.py with:

import os
import time

from azure.identity import DefaultAzureCredential
from azure.servicebus import NEXT_AVAILABLE_SESSION, ServiceBusClient
from azure.servicebus.exceptions import OperationTimeoutError

fully_qualified_namespace = os.environ["SERVICEBUS_FQNS"]

with ServiceBusClient(fully_qualified_namespace, DefaultAzureCredential()) as client:
    while True:
        try:
            with client.get_queue_receiver(
                queue_name="order-sessions",
                session_id=NEXT_AVAILABLE_SESSION,
                max_wait_time=5,
            ) as receiver:
                session_id = receiver.session.session_id
                print(f"Accepted session {session_id}")
                for message in receiver:
                    print(f"{message.session_id}: {str(message)}")
                    time.sleep(0.3)
                    receiver.complete_message(message)
        except OperationTimeoutError:
            print("No unlocked sessions remain.")
            break
Concurrency: Run a second processor.py process to accept another unlocked session. A session lock gives one receiver exclusive access to that order while preserving FIFO delivery within the session.
