Lab 5 - Scale orders from HTTP to storage

Build and test a complete serverless path through Python Functions, Service Bus, Event Grid, Logic Apps, and Table Storage.

90 minutesAzure FunctionsServerless integration

Case study: absorb a checkout surge

Flow: Azure Load Testing calls an HTTP-triggered Python Function. The Function sends an order to orders. When no receiver is active, Service Bus emits an Event Grid notification. A Logic App handles that notification, drains available messages with peek-lock, writes rows to Azure Table Storage, and completes each message.

flowchart LR
  ALT[Azure Load Testing] -- HTTP POST --> FN[Python Function producer]
  FN -- managed identity --> SB[Service Bus orders queue]
  SB -- queue has messages event --> EG[Event Grid]
  EG -- trigger --> LA[Logic App consumer]
  LA -- receive and complete --> SB
  LA -- upsert order --> TS[Table Storage]

Outcome: Load-test an HTTP endpoint, buffer accepted orders, wake a serverless consumer, and verify that every unique order becomes a table row.

1. Provision the supporting resources

Reuse rg-sb-workshop, the Service Bus namespace, and the orders queue from Lab 1. Do not create another namespace or queue. If orders is missing, return to Lab 1 and create it before continuing. Python Functions run on Linux. The commands use a Flex Consumption plan and a system-assigned managed identity.

Option A: Azure portal

  1. In the Azure portal, create a Storage account in rg-sb-workshop. Choose a unique name, the same region as the workshop namespace, Standard performance, and Locally-redundant storage (LRS).
  2. Open the storage account, select Access control (IAM) > Add role assignment, and assign your user account Storage Table Data Contributor. Then select Storage browser > Tables > Add table and create a table named Orders.
  3. Create a Function App in rg-sb-workshop. Select Flex Consumption, the same region, Python 3.11 on Linux, and the storage account you just created. Enter a globally unique app name such as func-sb-<unique>.
  4. Open the Function App, select Settings > Identity, turn the system-assigned identity On, and save.
  5. Open the Lab 1 Service Bus namespace and copy its Host name from the Overview page, for example sb-contoso-12345.servicebus.windows.net. Do not include https://, a trailing slash, or /orders.
  6. Return to the Function App. Under Settings > Environment variables, add SERVICEBUS_FQNS and paste that Host name as its value, then apply the change.
  7. Open the Service Bus namespace and select Access control (IAM) > Add role assignment. Assign Azure Service Bus Data Sender to the Function App's managed identity.

Replace function_app.py with the Python v2 programming model Function:

import json
import os

import azure.functions as func
from azure.identity import DefaultAzureCredential
from azure.servicebus import ServiceBusClient, ServiceBusMessage

app = func.FunctionApp(http_auth_level=func.AuthLevel.FUNCTION)
credential = DefaultAzureCredential()
service_bus_client = ServiceBusClient(os.environ["SERVICEBUS_FQNS"], credential)

@app.route(route="orders", methods=["POST"])
def submit_order(request: func.HttpRequest) -> func.HttpResponse:
    try:
        order = request.get_json()
        order_id = order["orderId"]
    except (ValueError, KeyError):
        return func.HttpResponse(
            json.dumps({"error": "A JSON orderId is required."}),
            mimetype="application/json",
            status_code=400,
        )

    with service_bus_client.get_queue_sender(queue_name="orders") as sender:
        sender.send_messages(ServiceBusMessage(
            json.dumps(order),
            message_id=order_id,
            subject="OrderCreated",
            content_type="application/json",
        ))

    return func.HttpResponse(
        json.dumps({"orderId": order_id}),
        mimetype="application/json",
        status_code=202,
    )

3. Configure Event Grid, Logic Apps, and Table Storage

  1. Create a Consumption Logic App named logic-sb-orders in rg-sb-workshop. Enable its system-assigned identity.
  2. Grant that identity Azure Service Bus Data Receiver on the namespace and Storage Table Data Contributor on the storage account.
  3. Start the workflow with When a resource event occurs. Select the subscription and resource group, set resource type to Microsoft.ServiceBus.Namespaces, select the workshop namespace, and choose event type Microsoft.ServiceBus.ActiveMessagesAvailableWithNoListeners.
  4. Add a condition that continues only when the Event Grid subject ends with /queues/orders. This prevents another queue's notification from starting this drain path.
  5. Add an Until loop controlled by a Boolean variable named queueEmpty, initially false. Set a finite timeout and iteration limit, such as five minutes and 500 iterations.
  6. Inside the loop, use the Service Bus managed-identity connection to Get messages from a queue (peek-lock) from orders, one message at a time.
  7. If no message is returned, set queueEmpty to true. Otherwise parse the JSON body, then use the Table Storage managed-identity connection to insert or replace an entity in Orders. Set PartitionKey to orders, RowKey to orderId, and store the message ID, amount, and received UTC time.
  8. Only after the table upsert succeeds, use the message lock token with Complete the message in a queue. Configure failures to abandon or dead-letter rather than completing unpersisted work.
$logicPrincipal = az resource show --resource-group $resourceGroup --resource-type Microsoft.Logic/workflows --name $logicApp --query identity.principalId -o tsv
az role assignment create --assignee $logicPrincipal --role "Azure Service Bus Data Receiver" --scope $namespaceId
az role assignment create --assignee $logicPrincipal --role "Storage Table Data Contributor" --scope $storageId
Important: Event Grid is a wake-up signal. It does not emit one event per Service Bus message and does not carry the message body. Stop all interactive receivers before testing, and make the Logic App drain until receive returns no message.

4. Run the load test

Create locustfile.py beside the Function project. Each request uses a unique order ID so Table Storage row counts remain meaningful.

import os
import uuid

from locust import HttpUser, between, task

class CheckoutUser(HttpUser):
    wait_time = between(0.1, 0.5)

    @task
    def submit_order(self):
        order_id = str(uuid.uuid4())
        self.client.post(
            "/api/orders",
            params={"code": os.environ["FUNCTION_KEY"]},
            json={"orderId": order_id, "amount": 125},
            name="POST /api/orders",
        )

Get the deployed endpoint and function key, then run a 20-user, 60-second smoke load locally. You can upload the same Locust script to Azure Load Testing and configure 20 users, a spawn rate of 5, and a 60-second duration.

python -m pip install locust

5. Verify the complete path

Compare successful Function requests, the queue's active-message count, Event Grid delivery metrics, Logic App run history, dead-letter count, and unique Table Storage row count. Increase users until the queue grows, then watch it drain after the load stops.

az servicebus queue show --resource-group $resourceGroup --namespace-name $namespace --name orders --query countDetails
az storage entity query --account-name $storage --table-name Orders --auth-mode login --query "length(items)" -o tsv

Expected result: HTTP calls return 202, Event Grid starts the Logic App, the queue returns to zero active messages with no dead letters, and the Orders table contains one row per unique accepted order ID.

Production extension: Use Premium Service Bus when you need isolated capacity and predictable latency; use private endpoints, zone redundancy where available, Application Insights, DLQ alerts, and idempotent storage writes.
