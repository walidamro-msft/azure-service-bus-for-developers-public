Lab 1 - Queue-based order intake

Create a Service Bus queue, then build separate producer and receiver Python applications.

60 minutesTwo Python appsQueues

Case study: protect checkout from traffic spikes

Contoso Shop must accept orders even when the fulfilment service is busy or temporarily offline. The checkout producer writes each order to a durable queue and returns immediately. A separate fulfilment receiver drains the queue at its own pace.

flowchart LR
  C[Checkout producer] -- OrderCreated --> Q[orders queue]
  Q -- competing consumer --> F[Fulfilment receiver]

Outcome: You will deploy the queue, build two independently runnable Python applications, send ten orders, and prove that the receiver settles every message.

1. Provision the queue

Use Standard so the same namespace can support topics and sessions in later labs. Basic is sufficient for a queue-only experiment; Premium is appropriate when production requires dedicated capacity, private endpoints, or predictable latency.

Option A: Azure portal

  1. Open the Create a Service Bus namespace page.
  2. Select your subscription, create or select rg-sb-workshop, enter a globally unique namespace such as sb-contoso-<unique>, choose a region, and select the Standard pricing tier.
  3. Select Review + create, then Create. When deployment finishes, open the namespace.
  4. Under Settings, open Shared access policies and disable Local Authentication. The applications use Microsoft Entra ID instead of access keys.
  5. Under Entities, select Queues and + Queue. Set the name to orders, set Max delivery count to 5, keep the other defaults, and select Create.
  6. Open Access control (IAM), select Add > Add role assignment, choose Azure Service Bus Data Owner, and assign it to your signed-in user.
  7. On the namespace Overview page, copy the hostname ending in .servicebus.windows.net. Set that value as SERVICEBUS_FQNS in every terminal used by this lab.

Option B: Azure CLI

az login
$suffix = Get-Random -Maximum 99999
$resourceGroup = "rg-sb-workshop"
$namespace = "sb-contoso-$suffix"

az group create --name $resourceGroup --location eastus
az servicebus namespace create --resource-group $resourceGroup --name $namespace --location eastus --sku Standard --disable-local-auth true
az servicebus queue create --resource-group $resourceGroup --namespace-name $namespace --name orders --max-delivery-count 5

$namespaceId = az servicebus namespace show --resource-group $resourceGroup --name $namespace --query id -o tsv
$userId = az ad signed-in-user show --query id -o tsv
az role assignment create --assignee $userId --role "Azure Service Bus Data Owner" --scope $namespaceId
$env:SERVICEBUS_FQNS = "$namespace.servicebus.windows.net"
Note: Complete either Option A or Option B, not both. RBAC may take several minutes to propagate. Both applications read SERVICEBUS_FQNS from the environment.

2. Create two Python applications

mkdir lab1-queue
cd lab1-queue
py -3.10 -m venv .venv
Set-ExecutionPolicy -Scope Process Bypass
.\.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip
python -m pip install azure-servicebus azure-identity
New-Item producer.py, receiver.py -ItemType File

The producer and receiver are separate processes. This separation demonstrates that the services do not need to start, stop, or scale together.

3. Build the checkout producer

Replace producer.py with:

import json
import os

from azure.identity import DefaultAzureCredential
from azure.servicebus import ServiceBusClient, ServiceBusMessage

fully_qualified_namespace = os.environ["SERVICEBUS_FQNS"]
credential = DefaultAzureCredential()

with ServiceBusClient(fully_qualified_namespace, credential) as client:
    with client.get_queue_sender(queue_name="orders") as sender:
        for number in range(1, 11):
            order_id = f"ORD-{number:03d}"
            body = json.dumps({"orderId": order_id, "product": "AZ-BOOK", "quantity": number})
            message = ServiceBusMessage(
                body,
                message_id=order_id,
                subject="OrderCreated",
                content_type="application/json",
            )
            sender.send_messages(message)
            print(f"Sent {order_id}")

4. Build the fulfilment receiver

Replace receiver.py with:

import json
import os

from azure.identity import DefaultAzureCredential
from azure.servicebus import ServiceBusClient

fully_qualified_namespace = os.environ["SERVICEBUS_FQNS"]
credential = DefaultAzureCredential()

with ServiceBusClient(fully_qualified_namespace, credential) as client:
    with client.get_queue_receiver(queue_name="orders", max_wait_time=20) as receiver:
        print("Receiver is waiting for orders. Press Ctrl+C to stop.")
        try:
            for message in receiver:
                order = json.loads(str(message))
                print(f"Received {order['orderId']} (delivery {message.delivery_count})")
                receiver.complete_message(message)
        except KeyboardInterrupt:
            print("Receiver stopped.")

5. Run and verify

Open two terminals in lab1-queue, set the same SERVICEBUS_FQNS value in each, and run:

# Terminal 1: start the receiver
.\.venv\Scripts\Activate.ps1
$env:SERVICEBUS_FQNS = "<namespace>.servicebus.windows.net"
python receiver.py

# Terminal 2: send ten orders
.\.venv\Scripts\Activate.ps1
$env:SERVICEBUS_FQNS = "<namespace>.servicebus.windows.net"
python producer.py

Expected result: The producer prints ten sent order IDs. The receiver prints those ten IDs and completes them. In the Azure portal, the queue's active message count returns to zero.

Challenge: Stop the receiver, run the producer again, and inspect the queue count. Restart the receiver and observe the durable backlog drain. Then run two receiver instances and confirm each order is processed by only one competing consumer.
