Working with Queues

Design and create point-to-point Service Bus queues, choose each entity setting deliberately, and define message contracts that work across programming languages.

Queue fundamentalsPortal, CLI, and BicepModule 6 of 11

What a Service Bus queue provides

A queue is a durable, point-to-point message entity inside a Service Bus namespace. Senders place messages on the queue without waiting for a receiver. Competing receivers then lock and process each message, so one successful receiver handles each delivery.

flowchart LR
    P1[Producer A] --> Q[(Orders queue)]
    P2[Producer B] --> Q
    Q --> C1[Worker 1]
    Q --> C2[Worker 2]
    Q -. failed or expired .-> D[Dead-letter subqueue]

Queues decouple producer and consumer availability, absorb bursts, and distribute work across competing consumers. Use a topic with subscriptions instead when multiple independent consumers each need their own copy.

Create a queue in the Azure portal

  1. Open the Service Bus namespace, select Queues, and then select + Queue.
  2. Choose the settings below before selecting Create. Settings shown after enabling a feature, such as the duplicate-detection window, must also be reviewed.
  3. After creation, open the queue and verify its status, message counts, and dead-letter count before connecting an application.
Creation-time decisions: duplicate detection, sessions, and Basic/Standard entity partitioning can't be enabled or disabled later. Replacing a queue requires a migration plan because entity recreation changes the operational endpoint.

Name

ValueEntity path
Limit1-260 characters
What it controls

The queue path used by senders, receivers, RBAC scopes, metrics, and forwarding settings.

How to choose

Use a stable lowercase business-purpose name such as orders-processing. Treat it as infrastructure, not a deployment instance name. Avoid secrets and values that change per release.

Max queue size

ValueCapacity
Allowed1, 2, 3, 4, or 5 GB; 80 GB when partitioned or Premium
What it controls

The maximum stored size of active, scheduled, deferred, and dead-lettered messages. New sends fail when the entity reaches quota.

How to choose

Estimate peak ingress minus processing over the longest expected outage, then include DLQ and growth headroom. Capacity is not retention: TTL still expires messages.

Max delivery count

ValueWhole number
Range1-2000; default 10
Why you need it

It prevents a poison message from being offered to workers forever. After the permitted PeekLock delivery attempts are exhausted, Service Bus removes the message from the active queue and places it in the DLQ with MaxDeliveryCountExceeded, allowing healthy messages to continue.

How the count starts

The broker sets DeliveryCount to 1 when a receiver gets the message for the first time. Completing the message removes it. Abandoning it, or allowing its lock to expire, makes it available again and increments the count for its next delivery. Explicitly dead-lettering it moves it immediately instead of waiting for the maximum.

Example: maximum is 3
  1. Worker A receives the message with DeliveryCount = 1. Processing fails and the worker abandons it.
  2. Worker B receives it with DeliveryCount = 2. Its lock expires before completion.
  3. A worker receives it with DeliveryCount = 3. If this attempt also fails, Service Bus moves the message to the DLQ rather than making a fourth normal delivery.
How to choose

Keep 10 as a practical starting point. Lower it when failures are usually deterministic, and raise it only when recovery from transient downstream failures genuinely needs more delivery attempts. SDK retries of a network operation are not automatically new message deliveries; this count protects the receive-and-process cycle.

Message time to live

ValueDays, hours, minutes, seconds
Portal starting value14 days
What it controls

The default lifetime applied when a message doesn't provide a shorter TimeToLive. Service Bus calculates ExpiresAtUtc = EnqueuedTimeUtc + TimeToLive. After that instant, the message is no longer eligible for normal receive.

What happens at expiration
  • If dead-letter on message expiration is disabled, Service Bus drops the expired message.
  • If it is enabled, Service Bus moves the message to the DLQ and sets DeadLetterReason to TTLExpiredException. TTL is not enforced inside the DLQ, so an operator or recovery process must remove the message.
Important timing details

Expiration is enforced lazily. An expired message might temporarily remain in the message count or appear in a peek result, but a normal receive doesn't return it. A message already locked by a receiver can pass its expiration time while being processed: completing it still succeeds, while abandoning it or allowing the lock to expire makes the TTL outcome apply.

How to choose

Set the business usefulness window, not an arbitrary retention target. Use minutes for time-sensitive commands and longer periods for recoverable backlog. A per-message TTL is capped by the entity default. Enable dead-letter-on-expiration only when expired work must be inspected, audited, or recovered.

Lock duration

ValueDuration
Range5 seconds to 5 minutes; default 1 minute
What it controls

How long a PeekLock receiver owns a delivery before another receiver can receive it. Completing after the lock expires fails.

How to choose

Set it above normal processing latency and use SDK lock renewal for legitimate long work. Do not set five minutes merely to hide slow or stuck handlers.

Auto-delete on idle

ValueOn/off plus duration
Minimum when enabled5 minutes
What it controls

Deletes the queue resource after no qualifying client operation occurs for the configured interval. This deletes the queue itself, not merely old messages. Any active, scheduled, deferred, or dead-lettered messages belonging to that queue are no longer available after deletion.

What resets the idle timer

Sending, scheduling, receiving, peeking, or updating the queue counts as activity. A receive call resets the timer even when it finds no messages. The timer therefore measures client operations, not whether the queue is empty or contains an old backlog.

Choose it when
  • A development or test run creates a temporary queue and abandoned runs need automatic cleanup.
  • An application creates a short-lived reply queue or per-job queue and can recreate it when needed.
  • The queue name and configuration are generated automatically, and losing any remaining messages after the idle interval is acceptable.
Leave it unchecked when
  • The queue is part of stable production topology, must retain backlog during a long outage, or is referenced by applications, RBAC assignments, alerts, dashboards, or forwarding rules.
  • Infrastructure doesn't automatically recreate the queue and restore its configuration.

Example: with 14 days configured, deletion becomes eligible only after 14 continuous days without a send, receive, peek, schedule, or management operation. Autoforwarding counts as receive activity, so a forwarding source normally doesn't become idle. An Azure Resource Manager CanNotDelete lock does not prevent this automatic deletion.

Duplicate detection

ValueOn/off plus history window
Window20 seconds to 7 days; default 10 minutes
What it controls

When enabled at creation, Service Bus records the MessageId of each accepted send for the configured history window. This history is independent of the message's current state: receiving and completing the original message does not remove its ID from duplicate-detection history. A repeat send within the window reports success to the sender but is immediately ignored and dropped. Basic tier doesn't support duplicate detection.

Example

A publisher sends MessageId = order-123/create at 10:00, and a worker completes it at 10:01. With a 10-minute window, another send with that ID before 10:10 is discarded even though the original is no longer in the queue. After the history window expires, Service Bus can accept that ID again.

How to choose

Enable it when publishers can retry ambiguous sends. Reuse the same MessageId for retries of the same logical message, and assign a new ID for a genuinely new operation. Keep the window close to the maximum expected send-retry interval because larger windows require more matching work and can reduce throughput.

What it doesn't prevent

Duplicate detection filters repeat sends; it doesn't prevent redelivery of an already accepted message after a receiver abandons it or loses its lock. Consumers should still be idempotent or record processed business IDs, especially because the broker forgets a MessageId after the history window.

Dead-letter on expiration

ValueOn/off
DefaultOff
What it controls

Moves expired messages to the DLQ with TTLExpiredException instead of removing them. DLQ messages do not expire automatically.

How to choose

Enable when operators need to inspect or recover expired work. Leave off when expiration itself is the intended cleanup and no audit is required. Monitor and drain the DLQ either way.

Partitioning

ValueOn/off
ScopeEntity-level in Basic/Standard; namespace-level in Premium
What it controls

Spreads a Basic/Standard queue across 16 brokers and stores for availability and capacity. Premium partitioning is selected when the namespace is created.

How to choose

Use for high-throughput or larger-capacity Basic/Standard queues. Avoid it for low-volume workloads that require the oldest global peek result. Partitioning can't be changed after creation.

Sessions

ValueOn/off
RequirementEvery message needs SessionId
Does it force FIFO?

Sessions provide FIFO delivery within each SessionId, not across the entire queue. A session receiver obtains an exclusive lock for one session, so messages with that ID are delivered to the same owning receiver in enqueue order. Messages in different sessions can be processed concurrently and can finish in any relative order.

Example

Set SessionId = order-123 on the Created, Paid, and Shipped events for one order. Service Bus delivers that sequence in order to the receiver that owns order-123, while order-456 can be handled by another receiver at the same time. To keep business side effects in order, configure the application to process one message at a time within each session and settle it before advancing.

How to choose

Enable sessions when related messages must be processed sequentially, or when a workflow needs exclusive per-key state. Good session keys include an order ID, account ID, or workflow instance ID. Using one shared SessionId for every message approximates whole-queue FIFO but removes most parallelism and creates a throughput bottleneck.

Important constraints

Every message sent to a session-enabled queue must have a nonempty SessionId, and receivers must use session-aware APIs. Sessions are supported in Standard and Premium tiers, can't be enabled or disabled after queue creation, and can't be used on an autoforwarding source.

Forward messages to queue/topic

ValueDestination entity name
ScopeSame namespace; maximum 4 hops
Why enable it?

Enable forwarding when every message arriving at this queue should be moved automatically to another queue or topic without deploying and operating a relay application. Service Bus performs the transfer inside the broker: the source becomes an ingress or chaining endpoint, while consumers receive from the destination.

Useful scenarios
  • Consolidation: forward several source queues or topic subscriptions into one central processing queue so one consumer pool handles the combined workload.
  • Stable producer endpoint: keep producers sending to an established source name while processing is centralized behind a different destination.
  • Broker-managed chaining: connect messaging stages when every message follows the same path and no application-level transformation or routing decision is required.
Example

web-orders, store-orders, and partner-orders can each forward to all-orders. Producers retain channel-specific send endpoints, but workers need to monitor and process only all-orders.

Leave it empty when

An application must inspect, transform, enrich, filter, or choose a destination based on message content. Also leave it empty when this queue needs its own receiver, because applications can't receive directly from a forwarding source.

Important constraints

The destination must already exist in the same namespace, Basic tier isn't supported, and a session-enabled entity can't be the source. Each forwarded message is a billable operation. A disabled, deleted, or full destination causes messages to enter the source transfer DLQ, which must be monitored and drained. Chains exceeding four forwarding hops are dead-lettered.

Create a queue with Azure CLI

The Azure Cloud Shell Bash example creates an existing namespace's production-style queue. Azure CLI durations use ISO 8601: PT1M is one minute, PT10M is ten minutes, and P14D is fourteen days.

resourceGroup="rg-servicebus-workshop"
namespaceName="sb-workshop-demo"
queueName="orders-processing"

az servicebus queue create \
  --resource-group "$resourceGroup" \
  --namespace-name "$namespaceName" \
  --name "$queueName" \
  --max-size 1024 \
  --max-delivery-count 10 \
  --default-message-time-to-live P14D \
  --lock-duration PT1M \
  --enable-duplicate-detection true \
  --duplicate-detection-history-time-window PT10M \
  --enable-dead-lettering-on-message-expiration true \
  --enable-partitioning false \
  --enable-session false

--max-size is in MB. Add --auto-delete-on-idle P1D only for an intentionally temporary queue, or --forward-to destination-name only after the destination exists.

Create a queue with Bicep

This resource targets an existing namespace and expresses the same queue contract as code. Keep stable topology in source control and review creation-time settings before deployment.

@description('Name of the existing Service Bus namespace.')
param namespaceName string

@description('Name of the queue to create.')
@minLength(1)
@maxLength(260)
param queueName string = 'orders-processing'

resource serviceBusNamespace 'Microsoft.ServiceBus/namespaces@2024-01-01' existing = {
  name: namespaceName
}

resource queue 'Microsoft.ServiceBus/namespaces/queues@2024-01-01' = {
  parent: serviceBusNamespace
  name: queueName
  properties: {
    maxSizeInMegabytes: 1024
    maxDeliveryCount: 10
    defaultMessageTimeToLive: 'P14D'
    lockDuration: 'PT1M'
    requiresDuplicateDetection: true
    duplicateDetectionHistoryTimeWindow: 'PT10M'
    deadLetteringOnMessageExpiration: true
    enablePartitioning: false
    requiresSession: false
  }
}

For a disposable queue, add autoDeleteOnIdle with a value of at least PT5M. For autoforwarding, add forwardTo or forwardDeadLetteredMessagesTo and declare a dependency on the destination entity. The same example is available as a standalone Bicep file.

Service Bus message anatomy

A Service Bus message is an envelope, not just a JSON string. The envelope contains the business payload, standard AMQP properties understood by Service Bus and other AMQP 1.0 clients, custom application properties, and broker-managed delivery metadata. A queue and a topic use the same message structure; only the destination and delivery model differ.

Body: the business payload

Body is an opaque AMQP value or sequence of bytes. It can contain JSON, text, XML, Avro, Protocol Buffers, or arbitrary binary data. Each SDK maps it to a language-specific type. Keep the payload self-contained and versioned, set ContentType so consumers know how to deserialize it, and validate both content type and schema before running business logic.

Message size budget: The body has no separate maximum. The encoded body, standard properties, and application properties together must fit the message limit: 256 KB in Basic and Standard; in Premium, 1 MB by default and up to 100 MB for a single AMQP message when the entity is configured for large messages. HTTP messages and all batches remain limited to 1 MB. Each individual property can occupy at most 32 KB, and all system plus application properties combined can occupy at most 64 KB. Keep useful headroom because AMQP encoding and broker properties also consume bytes.
{
  "orderId": "ORD-123",
  "status": "created",
  "occurredAtUtc": "2026-07-20T10:30:00Z"
}

The message envelope would identify this payload with ContentType = application/json and a stable contract name such as Subject = OrderCreated.v1. The language subtopics show the exact constructor and field syntax.

Properties set by the sender

MessageId

Data typeString
Size or rangeMaximum 128 characters.
Meaning

Identity of one logical message. Service Bus duplicate detection uses this value when enabled on the queue or topic.

Value strategy

Assign one ID when the message is first created. Reuse that exact ID every time you retry sending the same message. Assign a new ID only when you create a genuinely new command or event.

  • Same message, same ID: sending an OrderCreated event fails or times out, so the publisher retries. Both attempts use the original message ID.
  • New message, new ID: OrderCreated and a later OrderShipped event for the same order must have different message IDs.
  • Do not use only the entity ID: using ORD-123 for every message about that order would make distinct events look like duplicates.
  • Recommended: generate a UUID once and persist it with the outbox record. A deterministic value such as OrderCreated:ORD-123 is safe only when the business guarantees that event can occur once for that order.

CorrelationId

Data typeString
Size or rangeNo smaller field-specific quota is documented; the 32-KB property and 64-KB cumulative header limits apply.
Meaning

Connects related messages and operations without making them the same message.

Value strategy

Use a workflow, request, trace, or aggregate ID such as order:123. Propagate it through downstream messages. Do not use it as the idempotency key unless it uniquely identifies one operation.

Subject

Data typeString
Size or range32-KB individual-property limit; also counts toward the 64-KB header limit.
Meaning

Describes the message's business purpose. Older SDKs and documentation may call this property Label.

Value strategy

Use a stable contract name, optionally with a major version, such as OrderCreated.v1. Consumers should not infer the type from a language-specific class name.

ContentType

Data typeString
Size or range32-KB individual-property limit; keep it to a short MIME media type.
Meaning

MIME-style description of the body format.

Value strategy

Use values such as application/json or a vendor media type. This describes serialization, while Subject describes business meaning.

ApplicationProperties

Data typeMap<string, scalar>
Size or rangeEach property is at most 32 KB; all properties and system headers together are at most 64 KB.
Meaning

Small typed key/value metadata available to consumers and subscription filters without reading the body. Supported values include strings, Boolean and numeric primitives, characters, UUIDs, date/time values, URIs, durations, and streams. Binary values currently need Base64 string encoding because raw byte arrays are rejected by the service.

Value strategy

Put routing and operational facts here, such as region, tenantId, or schemaVersion. Keep names stable and values compact. Do not place secrets, large payload fragments, or high-cardinality diagnostics here.

SessionId

Data typeString
Size or rangeMaximum 128 characters.
Meaning

Groups related messages into an ordered, exclusive session on a session-enabled entity.

Value strategy

Use the aggregate or workflow key whose messages must be processed in order, such as order:123. Every message sent to a session-enabled entity must have a session ID.

PartitionKey

Data typeString
Size or range32-KB individual-property limit; also counts toward the 64-KB header limit.
Meaning

Keeps related messages together for partition-aware transfer. When sessions are used, it must match SessionId.

Value strategy

Set it only when your topology requires partition affinity or transactional grouping. Avoid a constant value because it creates a hot partition.

TimeToLive

Data typeDuration
Size or rangeCannot effectively exceed the entity's DefaultTimeToLive; Service Bus silently caps a longer value.
Meaning

Maximum time the message may remain available before it expires.

Value strategy

Derive it from business usefulness. Use a short TTL for stale commands and a longer one for recoverable work. Decide whether expired messages should be dead-lettered.

ScheduledEnqueueTime

Data typeUTC timestamp
Size or rangeNo independent size quota beyond the encoded property budget.
Meaning

Asks the broker to make the message available at a future UTC time.

Value strategy

Use for delayed delivery, not as a precise scheduler. Consumers must still handle late delivery and duplicates.

ReplyTo

Data typeString
Size or range32-KB property limit; an actual queue or topic path is limited to 260 characters.
Meaning

Names the entity where a response should be sent.

Value strategy

Use only for an intentional request/reply contract. Combine with CorrelationId so the requester can match the response.

ReplyToSessionId

Data typeString
Size or range32-KB individual-property limit; choose a compact session identifier.
Meaning

Specifies the SessionId to use on a reply sent to ReplyTo.

Value strategy

Use when multiple requesters share a session-enabled reply entity.

To

Data typeString
Size or range32-KB property limit; an entity path, when used, is limited to 260 characters.
Meaning

Logical destination metadata carried inside the envelope.

Value strategy

Service Bus currently reserves and ignores this property for broker routing. Use application properties for application-level routing; the sender determines the actual queue or topic.

Properties assigned by Service Bus

SequenceNumber

Data typeSigned 64-bit integer
Size or rangePartitioned entities encode the partition in the highest 16 bits.
What it tells the receiver

A broker-assigned number that identifies the enqueued message within the entity. It is useful for diagnostics and deferral, but it is not a cross-entity business ID.

EnqueuedTime / ExpiresAt

Data typeUTC timestamp
Size or rangeRepresented by the client language's date/time range.
What it tells the receiver

When Service Bus accepted the message and when its TTL expires. Use these broker timestamps when measuring queue age.

DeliveryCount

Data typeSigned 32-bit integer
Size or rangeOperationally bounded by the entity's configured MaxDeliveryCount before dead-lettering.
What it tells the receiver

How many delivery attempts have occurred. A high value signals repeated abandon operations or lock loss.

LockToken

Data typeString
Size or rangeOpaque broker value; do not parse or generate it.
What it tells the receiver

Identifies the current peek-lock ownership used to settle or defer the message.

LockedUntil

Data typeUTC timestamp
Size or rangeDetermined by the entity lock duration and any renewals.
What it tells the receiver

Settle before this instant or renew the lock while long-running work continues.

DeadLetterSource

Data typeString
Size or rangeEntity path metadata; subject to the encoded property budget.
What it tells the receiver

The source entity when a message is received through dead-letter forwarding.

Important: Duplicate detection at the broker and idempotent processing in the consumer solve different problems. Enable duplicate detection where appropriate, but also store processed MessageId values or enforce a business uniqueness constraint so a redelivered message cannot repeat side effects.

Receive and settle

Every supported SDK offers a receive client. Some also provide a dedicated processor abstraction; others attach callbacks to a receiver or let your code run the receive loop. The broker behavior is the same even when the class and option names differ.

Event-driven processing

Best forContinuous workers
Your code ownsMessage and error handlers
How it works

The SDK keeps receiving in the background and dispatches messages to your handler. Depending on the language, this is a processor object, a receiver subscription, or a loop built around the receiver.

Operational responsibility

Start and stop gracefully, bound concurrency, settle only after durable work succeeds, and record errors with enough entity and client context to diagnose failures.

Pull-based receiving

Best forExplicit receive control
Your code ownsThe receive loop and batch size
How it works

Your code requests one or more messages, iterates a receive stream, or peeks without changing message state. This model is useful for scheduled jobs, administrative tools, deferral, and dead-letter recovery.

Operational responsibility

Handle empty receives and cancellation, renew locks when processing can run long, settle through the receiver that obtained the message, and close the receiver during shutdown.

Settlement rule: in PeekLock mode, complete only after required side effects are durable. Abandon transient failures, defer work that must be resumed by sequence number, and dead-letter permanent failures with a reason and description.

Common processing controls

Use these decisions in every language. The language references below map each control to its exact SDK type and option name.

Settlement policy

DecisionAutomatic or explicit
Start withExplicit settlement
What it controls

Whether the SDK completes a message after a successful handler or your code explicitly chooses complete, abandon, defer, or dead-letter.

How to choose

Use explicit settlement when success depends on durable downstream work or when permanent failures need reasoned dead-lettering. Automatic completion is suitable only when a successful handler return always means the work is finished.

Handler concurrency

DecisionParallel work per process
Start withOne handler
What it controls

The maximum number of messages one process can handle at the same time. Some SDKs expose a processor option; others require application-managed tasks, threads, or workers.

How to choose

Potential concurrency is approximately replicas multiplied by handlers per replica. Increase gradually while monitoring CPU, memory, database pools, downstream limits, handler latency, and lock-lost failures. Concurrency greater than one does not preserve queue-wide order.

Lock renewal window

DecisionMaximum automatic renewal time
Start withAbove expected handler time
What it controls

How long the client keeps renewing a PeekLock message or session while processing continues. Entity lock duration still controls each individual lock interval.

How to choose

Use a finite window above the longest expected successful processing time. A very long window can hide stuck handlers; a short window can cause settlement to fail and the message to be redelivered.

Prefetch or local cache

DecisionMessages requested ahead
Start withDisabled or conservative
What it controls

How much work the client buffers locally to reduce receive latency. Not every SDK exposes a direct prefetch setting.

How to choose

Increase only for fast, steady workloads. PeekLock time is already running while a prefetched message waits locally, and every replica consumes additional memory. Keep the cache low for large messages, slow handlers, short locks, or bursty traffic.

Receive mode

DecisionPeekLock or ReceiveAndDelete
Start withPeekLock
What it controls

PeekLock requires settlement and supports retry. ReceiveAndDelete removes a message when it is transferred to the client.

How to choose

Use PeekLock for business workloads. Use ReceiveAndDelete only when message loss is acceptable because a client crash after transfer cannot return the message to the entity.

Message source

DecisionPrimary entity or subqueue
Start withPrimary entity
What it controls

Whether the receiver reads active messages, the dead-letter subqueue, or the transfer dead-letter subqueue.

How to choose

Use a separate, deliberate recovery path for dead-letter processing. Do not point the normal business worker at a dead-letter subqueue.

Client identifier

DecisionDiagnostic client name
Start withRole plus replica suffix
What it controls

A name included in selected client logs and errors. It does not affect routing, sessions, or message identity.

How to choose

Use a recognizable role and a unique host or replica suffix, such as orders-worker:instance-07, so operators can associate failures with one client instance.

Tune controls together: more concurrency raises downstream pressure, while deeper buffering raises memory use and the time locked messages may wait. Measure handler latency, active-message age, redelivery, lock-lost failures, downstream saturation, and memory before and after each change.

Common SDK object roles

Class names differ, but the roles below are consistent. Choose an event-driven processor or subscription for continuous workers and a pull receiver when your application must control each receive call.

flowchart LR
    C[Namespace client or builder] --> S[Sender]
    C --> P[Processor or subscription]
    C --> R[Pull receiver]
    C --> SR[Session-aware receiver]
    S --> B[Size-aware batch]
    A[Administration client] -. management plane .-> E[Queues, topics, subscriptions, and rules]

Namespace client or builder

CreatesSenders and receivers
Typical lifetimeApplication lifetime
Use it for

Authentication, transport, retry configuration, and creating entity-specific clients. Depending on the SDK, child clients share its AMQP connection or a builder creates clients that share connection resources.

Lifecycle guidance

Create during startup and reuse. Do not construct a new top-level client for every operation or message. Close it after processors, subscriptions, senders, and receivers have stopped.

Sender

Creates or sendsMessages and batches
Typical lifetimeLong-lived per entity
Use it for

Sending to a queue or topic, scheduling messages, and building size-aware batches.

Lifecycle guidance

Reuse a sender for regular publishing. Create another sender for a different entity rather than opening and closing one around each send.

Processor or subscription

OwnsBackground receive dispatch
Typical lifetimeWorker lifetime
Use it for

Continuous callback-driven processing with bounded concurrency, lock renewal, settlement policy, and an error path.

Lifecycle guidance

Register handlers before starting, keep it alive while the worker runs, stop accepting work during shutdown, await in-flight handlers, and then close it.

Pull receiver

OwnsOne entity receive link
Typical lifetimeReuse while consuming
Use it for

Single or batched receives, asynchronous iteration, peeking, deferred-message retrieval, manual lock renewal, explicit settlement, and subqueue access.

Lifecycle guidance

Reuse it for the same entity and receive mode. Settle locked messages through the receiver that delivered them and close it when that consuming workflow ends.

Session-aware receiver

OwnsOne or more session locks
Typical lifetimeWorker or accepted session
Use it for

Ordered, exclusive processing by session ID, including session-lock renewal and session state when the SDK supports it.

Lifecycle guidance

Use the SDK's session processor for continuous discovery or accept a specific/next session for pull-based control. Session concurrency and per-session handler concurrency are separate decisions.

Message and size-aware batch

RepresentsOutbound or received data
Typical lifetimeOne logical message or send
Use it for

An outbound message carries the body and publisher-assigned properties. A received message adds broker delivery metadata. A size-aware batch prevents an oversized group from being submitted.

Lifecycle guidance

Preserve message identity across send retries. Treat a received message as scoped to its current delivery and lock. Never ignore a batch API result that says the next message does not fit.

Administration client

ManagesEntities and runtime properties
Typical lifetimeAdministration workflow
Use it for

Creating, reading, updating, and deleting queues, topics, subscriptions, and rules, plus reading runtime counts.

Lifecycle guidance

Keep administration off the message hot path, use least privilege, and prefer reviewed infrastructure as code for stable topology.

Queue reliability patterns

  • Set MessageId for idempotency and duplicate handling.
  • Tune handler concurrency per host and downstream capacity.
  • Dead-letter permanent failures with reason codes.
  • Emit correlation IDs for traceability across services.
