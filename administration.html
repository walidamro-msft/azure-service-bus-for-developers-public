Administration and Operations

Run Service Bus as a production messaging system: inspect health, govern entities, manage capacity, and recover failed messages safely.

Hands-on~50 minModule 10 of 11

Build an operating model

Separate application data-plane access from administrative control. Producers normally need Data Sender, consumers need Data Receiver, and operators who manage entities need Data Owner. Scope every assignment to the smallest practical namespace or entity.

ResponsibilityQuestions to answerPrimary signal
AvailabilityCan clients connect, send, and receive?Server errors, successful requests
Flow healthIs work arriving faster than it is processed?Active messages, incoming/outgoing rates
Failure handlingWhy are messages failing, and can they be replayed safely?DLQ count, reason, message age
CapacityAre throttling, connections, size, or throughput limiting the workload?Throttled requests, namespace size, CPU in Premium

Inspect without changing state

Start every incident with read-only evidence. Runtime properties provide counts; peeking samples messages without locking or settling them.

using Azure.Identity;
using Azure.Messaging.ServiceBus;
using Azure.Messaging.ServiceBus.Administration;

string fullyQualifiedNamespace = "<namespace>.servicebus.windows.net";
string queueName = "orders-queue";
var credential = new DefaultAzureCredential();

var admin = new ServiceBusAdministrationClient(fullyQualifiedNamespace, credential);
QueueRuntimeProperties runtime = await admin.GetQueueRuntimePropertiesAsync(queueName);

Console.WriteLine($"Active: {runtime.ActiveMessageCount}");
Console.WriteLine($"Dead-letter: {runtime.DeadLetterMessageCount}");
Console.WriteLine($"Scheduled: {runtime.ScheduledMessageCount}");
Console.WriteLine($"Size: {runtime.SizeInBytes:N0} bytes");

await using var client = new ServiceBusClient(fullyQualifiedNamespace, credential);
await using ServiceBusReceiver receiver = client.CreateReceiver(queueName);
foreach (ServiceBusReceivedMessage message in await receiver.PeekMessagesAsync(10))
{
    Console.WriteLine($"{message.SequenceNumber}: {message.Subject}");
}

Checkpoint: Compare runtime counts with the Portal overview and explain why a peek does not reduce the active-message count.

Govern entity configuration

  • Manage namespace and entity definitions through Bicep or another reviewed infrastructure-as-code workflow.
  • Use the administration SDK for controlled runtime tooling and dynamic entity scenarios, not as an untracked replacement for infrastructure as code.
  • Review lock duration, default TTL, maximum delivery count, duplicate detection, sessions, partitioning, and forwarding before deployment because some settings cannot be changed later.
  • Record who owns each queue or subscription, its schema, its retention expectation, and its recovery runbook.

Pause or disable an entity only through an approved incident procedure. Deleting an entity deletes its messages and should never be the first response to a backlog.

Capacity and continuity

  • Know the quota and feature limits for the selected tier before load testing.
  • In Premium, monitor namespace CPU and memory and scale messaging units from measured demand.
  • Use batching and long-lived SDK clients; creating clients per message wastes connections and can trigger throttling.
  • Choose Geo-Disaster Recovery for metadata failover requirements and Geo-Replication when replicated message data is required and supported for the workload.

Scaling is not a substitute for consumer health. A growing backlog with flat outgoing throughput usually points to consumer capacity or failures first.

Geo-Replication

Geo-Replication keeps Service Bus metadata and message data synchronized between a primary region and a secondary region. Use the namespace hostname in your clients, not a hard-coded region URL, because Azure routes traffic to the current primary region after promotion.

Important: Promotion is customer initiated. Azure does not automatically fail over the namespace. In normal operation, the secondary region is a hot standby and you cannot read or write to it directly.
ScenarioWhat it meansWhat to expect
Regional outageThe primary region becomes unavailable or unreliable.Promote the secondary region manually so producers and consumers keep using the same namespace hostname against the new primary. During promotion, clients can see brief errors and reconnects.
Planned maintenanceYou need to service the primary region without stopping the workload.Promote the secondary first, then perform maintenance. This is the safest option when you want to control the timing of the cutover.
Region migrationYou want the namespace to run in a different region permanently.Enable Geo-Replication to the target region, wait for synchronization, then perform a planned promotion and optionally remove the old region afterward.
Long outage or lagThe primary is down for hours or the replica falls behind.The namespace keeps working after you promote, but replication is still only guaranteed for the data that made it to the secondary before promotion. If lag grows too large, the primary can throttle requests.
FailbackThe original primary returns after you already promoted the secondary.The old primary comes back as a secondary and resynchronizes. Do not assume it automatically contains every message that was in flight during the outage.

Two operating details matter most. First, promotion usually takes minutes rather than seconds, so plan for a short interruption while clients reconnect. Second, synchronous replication gives the strongest data assurance but increases latency because each commit must succeed in both regions; asynchronous replication keeps latency lower but allows lag, so the service may throttle once the lag limit is reached.

For a production runbook, rehearse promotion at least once, monitor the replication lag metrics, and decide in advance whether your team will use planned or forced promotion for a given incident.

Incident response sequence

  1. Confirm scope: namespace, entity, subscription, region, and affected client operation.
  2. Inspect Azure Service Health, metrics, runtime properties, and recent deployment or configuration changes.
  3. Classify the failure as authentication, network, quota/throttling, lock/settlement, application processing, or service availability.
  4. Mitigate with the least destructive action and preserve failed-message evidence.
  5. Verify recovery from both message-flow metrics and application traces.

Dead-letter recovery runbook

DLQ messages do not expire automatically. Alert on accumulation and define an owner. Before replay, inspect the reason, fix the cause, validate the payload, preserve identifying properties, and confirm that the consumer is idempotent.

Important: A replay sends a new message. Complete the original DLQ message only after the replacement send succeeds. Blind bulk replay can repeat the original outage.
