Topics and Subscriptions

Implement publish/subscribe messaging with independent subscribers and rule-based filtering.

Hands-on coding~40 minModule 7 of 11

Pub/sub model

A topic accepts one published message and fans out to multiple subscriptions, each with its own lifecycle and consumer group.

flowchart LR
    P[Publisher] --> T[(orders-topic)]
    T --> S1[(inventory-sub)]
    T --> S2[(billing-sub)]
    T --> S3[(notifications-sub)]

Create a topic in the Azure portal

  1. Open a Standard or Premium Service Bus namespace, select Topics, and then select + Topic. Basic namespaces do not support topics.
  2. Review every setting below before selecting Create. Enabling auto-delete or duplicate detection reveals an additional duration setting.
  3. After creation, open the topic, create at least one subscription, and verify that each subscription has the intended default rule.
Creation-time decisions: duplicate detection and Standard entity partitioning cannot be enabled or disabled later. Premium partitioning is selected at namespace creation. Recreating a topic also removes its subscriptions and their messages, so treat these settings as infrastructure decisions.

Name

ValueEntity path
Limit1-260 characters
What it controls

The topic path used by publishers, RBAC scopes, metrics, subscription configuration, and forwarding destinations.

How to choose

Use a stable lowercase business-event name such as orders-topic. Avoid secrets, version-specific deployment names, and consumer names because the topic represents the published stream.

Max topic size

ValueCapacity
Portal choices1, 2, 3, 4, or 5 GB; 80 GB when partitioned or Premium
What it controls

The topic's storage quota. New sends fail when the entity reaches quota. Topic storage includes the message copies retained by its subscriptions, so slow or unavailable subscribers contribute to capacity pressure.

How to choose

Estimate fan-out, the longest expected subscriber outage, dead-letter growth, and scheduled messages, then leave operating headroom. Capacity is not retention: message TTL still determines how long a copy remains useful.

Message time to live

ValueDays, hours, minutes, seconds
Portal starting value14 days
What it controls

The default lifetime for messages published without a shorter per-message TimeToLive. Each subscription copy expires when that lifetime elapses.

How to choose

Use the business usefulness window for the event. Confirm each subscription's expiration and dead-letter behavior because subscribers can have different recovery requirements.

Enable auto-delete on idle topic

ValueOn/off
Related fieldAuto-delete after idle for
What it controls

When enabled, Service Bus deletes the topic after the configured interval without a qualifying client or management operation. Deletion removes the topic, all subscriptions, and their messages rather than merely clearing old messages.

How to choose

Enable it only for temporary development, test, reply, or per-job topology that automation can recreate. Leave it off for stable production topics, and remember that the minimum enabled idle interval is 5 minutes.

Auto-delete after idle for

ValueDays, hours, minutes, seconds
Portal starting value14 days
What it controls

The continuous idle interval that must pass before an auto-delete-enabled topic is removed.

How to choose

Set it longer than expected pauses between publisher, subscriber, and management activity. Do not use auto-delete as message retention; use TTL for message lifetime.

Enable duplicate detection

ValueOn/off
AvailabilityStandard and Premium
What it controls

When enabled at creation, Service Bus records accepted MessageId values and discards repeat sends within the configured history window. Every subscription therefore receives at most one copy from those repeated publish attempts.

How to choose

Enable it when publishers retry ambiguous sends and can reuse the same deterministic MessageId. It filters duplicate sends, not redelivery to a subscriber, so consumers must still be idempotent.

Duplicate detection window

ValueDays, hours, minutes, seconds
Range20 seconds to 7 days; default 10 minutes
What it controls

How long Service Bus remembers each accepted MessageId. The window can be changed later only when duplicate detection was enabled at topic creation.

How to choose

Keep it close to the longest expected publisher retry interval. Larger windows retain more IDs and can reduce throughput.

Enable partitioning

ValueOn/off
ScopeEntity-level in Standard; namespace-level in Premium
What it controls

In Standard, it spreads the topic across 16 brokers and stores for availability, throughput, and an effective 80-GB entity capacity. Premium partitioning is inherited from the namespace and is not a topic-level choice.

How to choose

Use it for Standard workloads that need greater capacity or throughput. Avoid it for low-volume workloads that require a globally oldest peek result, and set a consistent partition key only when related events must stay together.

Support ordering

ValueOn/off
ScopeNonpartitioned topics
What it controls

Requests that Service Bus forward messages from a nonpartitioned topic to each subscription in send order. It defaults to off. On a partitioned topic, setting it to on has no effect.

How to choose

Enable it only on a nonpartitioned topic when subscribers depend on broker forwarding order. For business ordering through concurrent processing and retries, use session-enabled subscriptions and set SessionId to the aggregate or workflow key.

Publish code

ServiceBusSender sender = client.CreateSender("orders-topic");

var evt = new ServiceBusMessage("{\"orderId\":123,\"region\":\"us\"}")
{
    MessageId = "order-placed:123",
    CorrelationId = "order:123",
    Subject = "OrderPlaced.v1",
    ContentType = "application/json"
};
evt.ApplicationProperties["region"] = "us";
evt.ApplicationProperties["schemaVersion"] = 1;

await sender.SendMessageAsync(evt);

Message strategy for topics

A topic does not require a different message type. Publishers send the same ServiceBusMessage envelope used for a queue. Service Bus evaluates the topic's subscription rules and places a copy into every matching subscription. The original MessageId, CorrelationId, Subject, body, and application properties describe the same published event in every copy.

ConcernRecommended fieldTopic strategy
Event identityMessageIdAssign one immutable ID when the event is created and reuse it for publisher retries. Each subscriber sees that same event ID and can apply idempotency independently. Do not use only orderId when several event types or revisions can exist for one order.
Workflow tracingCorrelationIdUse the order, checkout, or distributed workflow ID. Several events may share one correlation ID while retaining different message IDs.
Contract selectionSubjectUse a stable event name such as OrderPlaced.v1. Subscribers can select the contracts they understand without parsing the body first.
Subscription routingApplicationPropertiesPromote only fields needed by filters, such as region, tenantId, or priority. Keep the authoritative value in the body when it is part of the business event.
Per-key orderingSessionIdOn a session-enabled topic, use the aggregate key for ordered handling. Subscriptions receive session-aware copies and their consumers process sessions independently.
CompatibilityContentType plus a versioned contractPrefer additive schema evolution. Introduce a new major subject or topic only for incompatible changes, and keep old subscribers running during migration.

Design filters deliberately

SQL filters can evaluate system and application properties, while correlation filters are optimized for exact matches. Prefer a small, documented set of routing properties. A subscription starts with a default rule that matches every message; remove or replace that rule when the subscription should receive only selected events.

  • Filter on stable envelope metadata, not values hidden inside the serialized body.
  • Treat property names, types, and casing as part of the event contract.
  • Plan for unmatched events and test every publisher event type against the complete rule set.
  • Avoid embedding consumer names in events. Publishers describe facts; subscription rules decide which consumers receive them.
Fan-out detail: Delivery state belongs to each subscription copy. Inventory and billing can therefore have different SequenceNumber, DeliveryCount, lock, completion, and dead-letter outcomes even though their copies retain the same publisher-assigned message identity and payload.

For the complete field-by-field envelope reference, including TTL, scheduling, reply-to, locks, and delivery metadata, see Service Bus message anatomy.

Subscription filters

az servicebus topic subscription rule create \
  --resource-group rg-sb-workshop \
  --namespace-name sb-workshop-demo \
  --topic-name orders-topic \
  --subscription-name billing-sub \
  --name OnlyUS \
  --filter-sql-expression "region = 'us'"

Use SQL and correlation filters to route messages to specialized handlers without custom broker code.

Consumer pattern

  • One processor instance per subscription queue path.
  • Independent scaling for each downstream capability.
  • Per-subscription DLQ operations and alerting.
